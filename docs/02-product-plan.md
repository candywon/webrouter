## 二、核心功能设计

### 2.1 一键部署（破局功能）

**目标：** 把 New-API + WebRouter 的部署从"10 步手动操作"变成"1 条命令"。

**方案：**

```bash
# 用户只需执行一条命令
curl -fsSL https://webrouter.dev/install.sh | bash
```

脚本自动完成：
1. 检测系统环境（OS、Docker、端口占用）
2. 拉取 WebRouter + New-API Docker 镜像
3. 生成 docker-compose.yml（含 New-API + WebRouter + Redis）
4. 启动所有服务
5. 打印访问地址和初始密码

**部署架构：**

```
docker-compose.yml
├── new-api        (端口 3000，API 网关核心)
├── webrouter      (端口 5000，管理后台)
├── redis          (端口 6379，缓存 + 会话)
└── nginx          (端口 80/443，反向代理 + SSL)
```

**核心指标：**

- Key 健康状态（正常/警告/失效）

### 2.3 渠道健康监控

**监控维度：**

| 检测项 | 频率 | 判定标准 | 动作 |
|--------|------|---------|------|
| Key 可用性 | 每 5 分钟 | 发送测试请求，能否正常返回 | 失效 → 告警 + 自动禁用 |
| 余额检查 | 每 30 分钟 | 调用官方余额接口 | 低于阈值 → 告警 |
| 延迟监控 | 实时 | P50/P95/P99 延迟统计 | P95 > 5s → 警告 |
| 错误率 | 实时 | 5xx / 429 / 403 比例 | 5min 内 > 10% → 告警 |
| 速率限制 | 实时 | 429 状态码频率 | 连续 3 次 429 → 降权 |

**健康状态模型：**

```
● 正常     — 所有检测通过
⚠ 警告     — 延迟偏高 / 余额低 / 偶发错误
✕ 失效     — 请求失败 / 余额耗尽 / 账号被封
⏸ 已禁用   — 管理员手动关闭
```

### 2.4 智能告警

**告警通道：**

| 通道 | 实现方式 | 适用场景 |
|------|---------|---------|
| 微信 | Server酱 / 企业微信 Webhook | 国内用户首选 |
| 邮件 | SMTP 发送 | 备用通道 |
| Webhook | 自定义 HTTP 回调 | 高级用户集成 |

**告警规则：**

```yaml
alerts:
  - name: key_failed
    condition: "key 连续 3 次测试失败"
    level: critical
    channels: [wechat, email]

  - name: balance_low
    condition: "余额 < 设定阈值"
    level: warning
    channels: [wechat]

  - name: error_rate_high
    condition: "5 分钟内错误率 > 10%"
    level: warning
    channels: [wechat, email]

  - name: usage_spike
    condition: "1 小时调用量 > 日均 3 倍"
    level: info
    channels: [email]
```

## 四、New-API 集成方案

### 4.1 为什么选 New-API 而非 One-API

| 对比项 | One-API | New-API |
|--------|---------|---------|
| UI | 旧版 | 现代化新 UI |
| 多语言 | 无 | 中/英/法/日 |
| 计费 | 基础 | 缓存计费、灵活定价 |
| 格式转换 | OpenAI only | OpenAI ⇄ Claude ⇄ Gemini |
| 支付 | 无 | EPay、Stripe |
| 登录 | 基础 | Discord/Telegram/OIDC |
| 思考模型 | 无 | 支持 o3/Claude thinking/Gemini thinking |
| Realtime API | 无 | 支持 |
| 数据兼容 | 原版 | 兼容 One-API 数据库 |
| Star | 3万+ | 3.2万+ |

**结论：** New-API 是 One-API 的全面增强版，功能覆盖更广，UI 更现代，且完全兼容 One-API 数据，用户从 One-API 迁移零成本。

### 4.2 集成方式

WebRouter 不修改 New-API 源码，通过以下方式集成：

1. **Docker 侧车模式：** WebRouter 和 New-API 在同一个 Docker Compose 中并行运行
2. **数据库共享：** WebRouter 直读 New-API 的数据库，获取渠道/用户/日志数据
3. **API 代理：** WebRouter 通过 New-API 的管理 API 进行操作（渠道增删、Key 管理等）
4. **日志采集：** WebRouter 从 New-API 的请求日志中提取监控数据

```
┌────────────────────────────────────────┐
│            Docker Compose              │
│                                        │
│  ┌──────────┐    ┌──────────────────┐  │
│  │ WebRouter │───▶│   New-API        │  │
│  │ (Flask)   │    │   (Go)           │  │
│  │  :5000    │    │   :3000          │  │
│  └─────┬─────┘    └────────┬─────────┘  │
│        │                   │            │
│        │  ┌────────────────┘            │
│        │  │ 共享数据库                    │
│        ▼  ▼                              │
│  ┌──────────┐    ┌──────────────────┐   │
│  │  Redis   │    │  MySQL/SQLite    │   │
│  │  :6379   │    │  :3306          │   │
│  └──────────┘    └──────────────────┘   │
│                                        │
│  ┌──────────────────────────────────┐  │
│  │  Nginx (:80/:443)                │  │
│  │  /        → WebRouter (管理面板)   │  │
│  │  /api/*   → New-API (API 代理)    │  │
│  │  /v1/*    → New-API (AI API)      │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘
```
