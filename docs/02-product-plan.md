# 产品方案文档

## 一、产品定位

**一句话：** WebRouter 是 AI 中转站的"建站与管理工具"——你带资源，我给工具。

**架构全景：**

```
┌─────────────────────────────────────────────────┐
│                用户自备资源层                      │
│   API Key / 订阅账号 / 逆向渠道 / 企业合同         │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│           Provider 抽象层（统一纳管）              │
│                                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 直连官方  │ │ 聚合平台  │ │ 自建网关          │  │
│  │ OpenAI   │ │ AnyRoute │ │ LiteLLM / 自研   │  │
│  │ Claude   │ │ OhMyGPT  │ │ wr-proxy 自带    │  │
│  │ Gemini   │ │ ...      │ │ ...              │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│           WebRouter（管理增强层）                  │
│   统一监控 · 智能调度 · 告警 · 计费 · 团队 · 对接  │
│                                                   │
│  ┌─────────────────┐   ┌──────────────────────┐  │
│  │ Flask 管理后台   │   │ wr-proxy Go 代理网关  │  │
│  │  CRUD/面板/告警  │◄─►│  路由/脱敏/重试/计量   │  │
│  │  共享 SQLite     │   │  二进制流支持         │  │
│  └─────────────────┘   └──────────────────────┘  │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│             终端工具 / IDE 插件                    │
│   Claude Code · Codex · OpenClaw · Hermes · ...  │
└─────────────────────────────────────────────────┘
```

**核心优势：** WebRouter 自带 wr-proxy 高性能 Go 代理网关，不再依赖任何第三方网关。Provider 抽象层让用户把散落在各处的 API 源（官方直连、聚合平台、自建网关）全部注册进来，一个面板看全。

## 二、核心功能设计

### 2.1 一键部署

**目标：** 把 wr-proxy + WebRouter 的部署从"多步手动操作"变成"1 条命令"。

**方案：**

```bash
# 用户只需执行一条命令
curl -fsSL https://webrouter.dev/install.sh | bash
```

脚本自动完成：
1. 检测系统环境（OS、Go、Python、端口占用）
2. 编译 wr-proxy Go 代理网关
3. 安装 Python 依赖 + 创建虚拟环境
4. 生成配置文件和启动脚本
5. 启动 wr-proxy + Flask 双进程

**部署架构：**

```
start.py
├── wr-proxy     (端口 5051，Go 代理网关)
└── webrouter    (端口 5050，Flask 管理后台)
```

### 2.3 Provider 聚合管理（差异化功能）

**这是 WebRouter 区别于所有竞品的核心功能。** 现有工具只能管理单一数据源，没有任何产品能跨源统一管理。

#### 2.3.1 Provider 类型定义

|| 类型 | 标识 | 说明 | 典型例子 | 数据能力 |
|------|------|------|---------|---------|
| 直连 | `direct` | 直接调用大模型官方 API | OpenAI、Claude、Gemini | 仅健康+延迟 |
| 聚合 | `aggregate` | 第三方聚合中转平台 | AnyRoute、OhMyGPT、API2D | 健康+延迟+计费 |
| 自建 | `litellm` | 自建 LiteLLM 代理 | 用户自己部署的 LiteLLM | 健康+延迟+模型列表 |
| 自建 | `custom` | 自研/其他网关 | 用户自建的任何 OpenAI 兼容网关 | 仅健康+延迟 |

#### 2.3.2 Provider 注册流程

用户添加数据源只需 3 步：

```
1. 选择类型     →  direct / aggregate / litellm / custom
2. 填写连接信息  →  名称、Base URL、API Key（直连/聚合只需这3项）
3. 可选配置     →  关联模型、检测间隔、备注标签
```

#### 2.3.4 不同 Provider 类型的数据获取策略

|| 能力 | direct | aggregate | litellm | custom |
|------|--------|-----------|---------|--------|
| 健康检测 | ✅ HTTP测试 | ✅ HTTP测试 | ✅ HTTP测试 | ✅ HTTP测试 |
| 延迟监控 | ✅ | ✅ | ✅ | ✅ |
| 模型列表 | ❌ | ❌ | ✅ /v1/models | ❌ |
| 成本分析 | 手动录入 | 手动录入 | 手动录入 | 手动录入 |

**关键设计：** 所有类型均做健康检测+延迟监控，纳入统一告警和面板。wr-proxy 作为内置网关直接提供完整的渠道、用户、日志、计费数据。

#### 2.3.5 跨 Provider 智能调度（V1.0）

当用户通过 WebRouter 的统一入口调用 API 时，wr-proxy 智能调度引擎会：

1. **可用性优先** — 排除健康状态为 dead/rate_limited 的 Provider
2. **延迟优先** — 在可用 Provider 中选延迟最低的
3. **成本优先** — 同模型同延迟时选成本更低的源
4. **权重调度** — 管理员可设置 Provider 权重（如主力源 80%，备用源 20%）
5. **故障切换** — 单个 Provider 整体不可用时秒级切到备用源

### 2.4 渠道健康监控

**监控维度：**

| 检测项 | 频率 | 判定标准 | 动作 |
|--------|------|---------|------|
| Key 可用性 | 每 5 分钟 | 发送测试请求，能否正常返回 | 失效 → 告警 + 自动禁用 |
| 余额检查 | 每 30 分钟 | 调用官方余额接口 | 低于阈值 → 告警 |
| 延迟监控 | 实时 | P50/P95/P99 延迟统计 | P95 > 5s → 警告 |
| 错误率 | 实时 | 5xx / 429 / 403 比例 | 5min 内 > 10% → 告警 |
| 速率限制 | 实时 | 429 状态码频率 | 连续 3 次 429 → 降权 |

**健康状态模型：**

```
● 正常     — 所有检测通过
⚠ 警告     — 延迟偏高 / 余额低 / 偶发错误
✕ 失效     — 请求失败 / 余额耗尽 / 账号被封
⏸ 已禁用   — 管理员手动关闭
```

## 四、wr-proxy 集成方案

### 4.1 架构设计

WebRouter 的核心创新在于 **Provider 抽象层 + wr-proxy 自研网关**：不绑定任何第三方网关，所有 API 源统一为 Provider 概念，由 wr-proxy 提供高性能代理能力。

```
WebRouter 的 Provider 层级结构：

  Provider（数据源）
  │
  ├── direct（直连官方）
  │   ├── 配置：name, base_url, api_key, models[]
  │   ├── 检测：发送 HTTP 测试请求
  │   └── 数据：仅 health + latency
  │
  ├── aggregate（聚合平台）
  │   ├── 配置：name, base_url, api_key, models[], pricing_info
  │   ├── 检测：发送 HTTP 测试请求
  │   └── 数据：health + latency + 手动录入成本
  │
  ├── litellm（自建 LiteLLM）
  │   ├── 配置：name, base_url, master_key
  │   ├── 检测：HTTP 测试 + /v1/models 获取模型列表
  │   └── 数据：health + latency + models[]
  │
  └── custom（自定义网关）
      ├── 配置：name, base_url, api_key, health_endpoint
      ├── 检测：自定义健康检测端点
      └── 数据：health + latency（可扩展）
```

wr-proxy 内置能力：
- 智能路由（6 维复杂度评分：输入长度、对话轮数、代码含量、工具调用、关键词、系统提示）
- 请求脱敏（正则引擎，流式/非流式均支持）
- 自动重试（SHA256 请求哈希去重 + 冷却 + 截断响应重试）
- 成本计量（定价表 → cost_cents 写入请求日志）
- 二进制流支持（audio/image 端点，跳过 JSON 解析链路）
- 多策略路由（smart / priority / round_robin / least_latency / cost_first）

### 4.2 为什么选择自研 wr-proxy

|| 对比项 | 依赖第三方网关 | wr-proxy 自研 |
|--------|--------------|-------------|
| 依赖风险 | 上游大版本变更可能破坏兼容 | 自主可控，按需演进 |
| 部署复杂度 | 需额外管理 sidecar 进程 | 内置集成，一键部署 |
| 数据一致性 | 需读第三方 DB schema | 共享 SQLite，无兼容层 |
| 二进制支持 | 通常只处理 JSON/文本 | 原生支持 audio/image 端点 |
| 智能路由 | 基础权重/轮询 | 6 维评分 + 多策略 |
| 脱敏能力 | 无 | 内置正则引擎 + 流式恢复 |
| 重试机制 | 简单重试 | SHA256 去重 + 冷却 + 截断重试 |

**结论：** wr-proxy 是 WebRouter 的内置核心网关，不是第三方依赖。用户可以只用 wr-proxy，也可以额外注册外部 Provider 做聚合管理。

### 4.3 集成方式

WebRouter 不修改任何上游源码，通过以下方式集成外部 Provider：

1. **Provider 注册：** 用户在面板中注册 API 源（直连/聚合/自建），填写连接信息
2. **统一检测：** 不同 Provider 类型走不同的健康检测逻辑，结果归一化
3. **数据读取：** 外部 Provider 仅做 HTTP 检测；wr-proxy 作为内置网关直接提供完整数据
4. **统一展示：** 所有 Provider 的健康状态、延迟、成本在同一个面板展示

```

│  │  │dir  │ │agg  │ │lite │ │custom│         │ │
```
