# 产品方案文档

## 一、产品定位

**一句话：** WebRouter 是 AI 中转站的"建站与管理工具"——你带资源，我给工具。

**与竞品的关系：**

```
┌─────────────────────────────────────────────────┐
│                用户自备资源层                      │
│   API Key / 订阅账号 / 逆向渠道 / 企业合同         │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│           Provider 抽象层（统一纳管）              │
│                                                   │
│  ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
│  │ 直连官方  │ │ 聚合平台  │ │ 自建网关          │  │
│  │ OpenAI   │ │ AnyRoute │ │ New-API / LiteLLM│  │
│  │ Claude   │ │ OhMyGPT  │ │ One-API / 自研   │  │
│  │ Gemini   │ │ ...      │ │ ...              │  │
│  └──────────┘ └──────────┘ └──────────────────┘  │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│           WebRouter（管理增强层）                  │
│   统一监控 · 智能调度 · 告警 · 计费 · 团队 · 对接  │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│             终端工具 / IDE 插件                    │
│   Claude Code · Codex · OpenClaw · Hermes · ...  │
└─────────────────────────────────────────────────┘
```

**核心升级：** 之前 WebRouter 只能管理 New-API 这一个入口，现在通过 Provider 抽象层，用户可以把散落在各处的 API 源（官方直连、聚合平台、自建网关）全部注册进来，一个面板看全。

## 二、核心功能设计

### 2.1 一键部署（破局功能）

**目标：** 把 New-API + WebRouter 的部署从"10 步手动操作"变成"1 条命令"。

**方案：**

```bash
# 用户只需执行一条命令
curl -fsSL https://webrouter.dev/install.sh | bash
```

脚本自动完成：
1. 检测系统环境（OS、Docker、端口占用）
2. 拉取 WebRouter + New-API Docker 镜像
3. 生成 docker-compose.yml（含 New-API + WebRouter + Redis）
4. 启动所有服务
5. 打印访问地址和初始密码

**部署架构：**

```
docker-compose.yml
├── new-api        (端口 3000，API 网关核心)
├── webrouter      (端口 5000，管理后台)
├── redis          (端口 6379，缓存 + 会话)
└── nginx          (端口 80/443，反向代理 + SSL)
```

### 2.3 Provider 聚合管理（差异化功能）

**这是 WebRouter 区别于所有竞品的核心功能。** 现有工具只能管理单一数据源（New-API 管自己的渠道、LiteLLM 管自己的代理），没有任何产品能跨源统一管理。

#### 2.3.1 Provider 类型定义

|| 类型 | 标识 | 说明 | 典型例子 | 数据能力 |
|------|------|------|---------|---------|
| 直连 | `direct` | 直接调用大模型官方 API | OpenAI、Claude、Gemini | 仅健康+延迟 |
| 聚合 | `aggregate` | 第三方聚合中转平台 | AnyRoute、OhMyGPT、API2D | 健康+延迟+计费 |
| 自建 | `newapi` | 自建 New-API 实例 | 用户自己部署的 New-API | 完整（渠道+用户+日志+计费） |
| 自建 | `oneapi` | 自建 One-API 实例 | 用户自己部署的 One-API | 完整（渠道+用户+日志） |
| 自建 | `litellm` | 自建 LiteLLM 代理 | 用户自己部署的 LiteLLM | 健康+延迟+模型列表 |
| 自建 | `custom` | 自研/其他网关 | 用户自建的任何 OpenAI 兼容网关 | 仅健康+延迟 |

#### 2.3.2 Provider 注册流程

用户添加数据源只需 3 步：

```
1. 选择类型     →  direct / aggregate / newapi / oneapi / litellm / custom
2. 填写连接信息  →  名称、Base URL、API Key（直连/聚合只需这3项）
3. 可选配置     →  关联模型、检测间隔、备注标签
```

**智能识别：** 如果用户输入的 Base URL 是已知的 New-API/One-API 实例（检测 `/api/status` 端点），自动切换为对应类型并获取更多数据。

#### 2.3.5 跨 Provider 智能调度（V1.0）

当用户通过 WebRouter 的统一入口调用 API 时，智能调度引擎会：

1. **可用性优先** — 排除健康状态为 dead/rate_limited 的 Provider
2. **延迟优先** — 在可用 Provider 中选延迟最低的
3. **成本优先** — 同模型同延迟时选成本更低的源
4. **权重调度** — 管理员可设置 Provider 权重（如主力源 80%，备用源 20%）
5. **故障切换** — 单个 Provider 整体不可用时秒级切到备用源

### 2.4 渠道健康监控

**监控维度：**

| 检测项 | 频率 | 判定标准 | 动作 |
|--------|------|---------|------|
| Key 可用性 | 每 5 分钟 | 发送测试请求，能否正常返回 | 失效 → 告警 + 自动禁用 |
| 余额检查 | 每 30 分钟 | 调用官方余额接口 | 低于阈值 → 告警 |
| 延迟监控 | 实时 | P50/P95/P99 延迟统计 | P95 > 5s → 警告 |
| 错误率 | 实时 | 5xx / 429 / 403 比例 | 5min 内 > 10% → 告警 |
| 速率限制 | 实时 | 429 状态码频率 | 连续 3 次 429 → 降权 |

**健康状态模型：**

```
● 正常     — 所有检测通过
⚠ 警告     — 延迟偏高 / 余额低 / 偶发错误
✕ 失效     — 请求失败 / 余额耗尽 / 账号被封
⏸ 已禁用   — 管理员手动关闭
```

## 四、Provider 集成方案

### 4.1 Provider 体系设计

WebRouter 的核心创新在于 **Provider 抽象层**：不绑定任何单一网关，而是把所有 API 源统一为 Provider 概念。

```
WebRouter 的 Provider 层级结构：

  Provider（数据源）
  │
  ├── direct（直连官方）
  │   ├── 配置：name, base_url, api_key, models[]
  │   ├── 检测：发送 HTTP 测试请求
  │   └── 数据：仅 health + latency
  │
  ├── aggregate（聚合平台）
  │   ├── 配置：name, base_url, api_key, models[], pricing_info
  │   ├── 检测：发送 HTTP 测试请求
  │   └── 数据：health + latency + 手动录入成本
  │
  ├── newapi（自建 New-API）
  │   ├── 配置：name, base_url, admin_token, db_uri
  │   ├── 检测：HTTP 测试 + DB 直读渠道状态
  │   └── 数据：完整（channels, tokens, users, logs, costs）
  │
  ├── oneapi（自建 One-API）
  │   ├── 配置：name, base_url, admin_token, db_uri
  │   ├── 检测：HTTP 测试 + DB 直读渠道状态
  │   └── 数据：完整（channels, tokens, users, logs）
  │
  ├── litellm（自建 LiteLLM）
  │   ├── 配置：name, base_url, master_key
  │   ├── 检测：HTTP 测试 + /v1/models 获取模型列表
  │   └── 数据：health + latency + models[]
  │
  └── custom（自定义网关）
      ├── 配置：name, base_url, api_key, health_endpoint
      ├── 检测：自定义健康检测端点
      └── 数据：health + latency（可扩展）
```

### 4.2 为什么仍以 New-API 为推荐 Provider

|| 对比项 | One-API | New-API |
|--------|---------|---------|
| UI | 旧版 | 现代化新 UI |
| 多语言 | 无 | 中/英/法/日 |
| 计费 | 基础 | 缓存计费、灵活定价 |
| 格式转换 | OpenAI only | OpenAI ⇄ Claude ⇄ Gemini |
| 支付 | 无 | EPay、Stripe |
| 登录 | 基础 | Discord/Telegram/OIDC |
| 思考模型 | 无 | 支持 o3/Claude thinking/Gemini thinking |
| Realtime API | 无 | 支持 |
| 数据兼容 | 原版 | 兼容 One-API 数据库 |
| Star | 3万+ | 3.2万+ |

**结论：** New-API 是 One-API 的全面增强版，推荐用户优先使用。但 WebRouter 不强制绑定，用户可以只用直连模式，不用任何自建网关。

### 4.3 集成方式

WebRouter 不修改任何上游源码，通过以下方式集成：

1. **Provider 注册：** 用户在面板中注册 API 源（直连/聚合/自建），填写连接信息
2. **统一检测：** 不同 Provider 类型走不同的健康检测逻辑，结果归一化
3. **数据读取：** `newapi`/`oneapi` 类型直读数据库；其他类型仅做 HTTP 检测
4. **统一展示：** 所有 Provider 的健康状态、延迟、成本在同一个面板展示

```
┌─────────────────────────────────────────────────┐
│                 WebRouter 核心                    │
│                                                   │

│  │          Provider Manager                   │ │
│  │  ┌─────┐ ┌─────┐ ┌─────┐ ┌──────┐ ┌────┐ │ │
│  │  │ P-1 │ │ P-2 │ │ P-3 │ │ P-4  │ │... │ │ │
│  │  │dir  │ │agg  │ │new  │ │lite  │ │    │ │ │
│  │  └──┬──┘ └──┬──┘ └──┬──┘ └──┬───┘ └────┘ │ │
│  └─────┼───────┼───────┼───────┼──────────────┘ │

│  │         Health Checker (统一检测)            │  │
│  └─────┬───────┬───────┬───────┬──────────────┘  │
```
