# 企业知识库 — 核心难点深度拆解

> 版本：v1.0 | 日期：2026-05-14

---

## 难点一：知识识别 — 不能靠关键词

### 问题本质

关键词匹配的是"话题"，不是"知识密度"。

```
"帮我查一下合同到期时间"       → 关键词"合同"命中 → 但这是查询，不是知识
"这份合同的违约金条款和行业惯例有冲突" → 同样命中"合同" → 这才是知识

"Q3毛利率3.2%"               → 关键词啥也匹配不到 → 但这是高价值数据
"分析一下毛利率"              → 关键词匹配到了 → 但这只是问题，不是知识
```

知识 vs 非知识的分界线：

**知识 = 包含确定性的信息产出（结论、数据、规则、判断）**

### 解决方案：多级漏斗

```
所有API对话 (100%)
      │
      ▼ 第1级：信号筛选（规则，零成本）
      │ 丢弃：短问答(<3轮且<200字) / 纯编程(code>60%) / 明确无知识意图
      │ 预计过滤：~80%
      │
      ▼ 第2级：知识密度评估（低成本LLM，批量处理）
      │ 输入：对话全文（每批20条）
      │ 输出：has_knowledge(bool) + knowledge_type + confidence
      │ 模型：qwen3-coder-flash（最廉价）
      │ 成本：约0.01元/100条
      │ 预计过滤：~60%（剩余的40%有知识）
      │
      ▼ 第3级：知识提取（中成本LLM，逐条处理）
         只处理第2级通过的条目
         按类型分流：factual → 严格锚定 / analytical+procedural → LLM提炼
```

### 第1级规则细节

```go
// 应该丢弃的
- 单轮短问答：turn_count < 3 && response_length < 200
- 纯编程类：code_block_ratio > 0.6
- 翻译类：prompt包含"翻译"/"translate"且response是纯翻译
- 改写类：prompt包含"改写"/"润色"/"rewrite"且无实质性信息产出
- 简单查询：prompt包含"是什么"/"什么是"且response < 100字

// 应该保留的
- 包含具体数字/日期/金额的对话
- 包含"结论"/"建议"/"分析"等判断词的对话
- 超过5轮的长对话（大概率涉及业务决策）
- 明确涉及公司业务流程的对话
```

### 第2级 Prompt（知识密度评估）

```
你是知识密度评估引擎。判断以下对话是否包含值得保存的企业知识。

【判断标准】
包含知识的对话：出现具体数据、结论、规则、流程、判断
不包含知识的对话：纯查询、编程、翻译、闲聊、改写

【输出格式】严格输出JSON，不要其他内容
{
  "has_knowledge": true/false,
  "knowledge_type": "data|conclusion|rule|none",
  "confidence": 0.0-1.0,
  "reason": "简短判断理由"
}

对话内容：
---
{conversation}
---
```

### 第3级 Prompt（知识提取 — factual 严格模式）

```
你是企业数据提取引擎。从以下对话原文中提取事实数据。

【铁律】
1. value字段必须从原文逐字复制，禁止任何改写、换算、推断
2. 如果原文说"收窄3.2个百分点至18.7%"，value只能写"18.7%"
3. 如果数据不确定或有歧义，certainty标为uncertain
4. original_sentence必须包含该数据的完整原句
5. 禁止计算、换算、四舍五入、补零
6. 原文中没有的数据，绝对不能出现在value中

对话原文：
---
{conversation}
---

输出JSON：
{
  "data_points": [
    {
      "metric": "指标名称",
      "value": "从原文逐字复制的值",
      "period": "时间范围",
      "unit": "单位",
      "original_sentence": "包含此数据的原文完整句子",
      "certainty": "certain 或 uncertain"
    }
  ],
  "has_no_data": false
}
```

## 难点二：部门归类 — 分类体系必须能随业务演化

### 传统方案的问题

```
年初：销售部、售后部、财务部
年中：新增跨境电商组（归销售还是独立？）
年末：跨境电商变成事业部，下面又有自己的法务和财务
```

如果预设分类树，每次组织变动都要手动调整。而且新业务方向在出现之前无法预见。

### 解决方案：双层归类

#### 第1层：部门归属（由 Token 确定，100%准确）

```
Token "张三-财务部"  → 归属财务部
Token "李四-销售部"  → 归属销售部
Token "王五-电商组"  → 归属电商组
```

不需要 LLM 猜，Token 天然携带部门信息。设置方式：

- 管理员在创建/编辑 Token 时填写 `knowledge_department`
- 也可从 LDAP/钉钉等企业系统同步

#### 第2层：业务域（LLM 提取 + 管理员确认，可演化）

初始预设8个通用域：

| domain_code | domain_name | 说明 |
|-------------|-------------|------|
| legal | 法务合规 | 合同、法规、纠纷 |
| finance | 财务审计 | 预算、报表、税务 |
| hr | 人力资源 | 招聘、薪酬、绩效 |
| admin | 行政办公 | 考勤、采购、制度 |
| sales | 销售商务 | 客户、报价、竞标 |
| service | 售后客服 | 投诉、维修、退换 |
| tech | 技术研发 | 架构、代码规范、部署 |
| strategy | 战略决策 | 规划、市场、竞争 |

LLM 提取时允许返回已有域之外的新标记：

```json
{
  "domain_code": "cross_border_ecommerce",  // 全新的域
  "domain_name": "跨境电商",
  "is_new_domain": true
}
```

新域自动入库，状态为 `pending`：

- 当 pending 域积累超过 20 条知识 → 管理后台弹出提示
- 管理员操作：确认激活 / 合并到已有域 / 忽略并删除
- 确认后自动加入后续 LLM 提取的可选域列表

**关键：分类体系是从数据中生长出来的，不是人为预设的。**

### 业务域演化状态机

```
[LLM发现新域]
      │
      ▼
   pending
      │
      ├── 管理员确认 → active
      │                    │
      │                    └── 后续知识自动归入
      │
      ├── 合并到已有域 → merged
      │                    │
      │                    └── 已有知识迁移 + 后续知识自动归入目标域
      │
      └── 忽略 → 删除
                     │
                     └── 相关知识标记为domain="unclassified"
```

## 难点三：数据类知识零幻觉

### 问题本质

LLM 提炼知识时，对数据的"理解"和"转述"会产生微妙但致命的错误：

```
原文："Q3毛利率同比收窄3.2个百分点至18.7%"
LLM提取："Q3毛利率下降3.2%"           ← 错误！收窄3.2个百分点≠下降3.2%
LLM提取："Q3毛利率为3.2%"             ← 严重错误！把百分点当绝对值
LLM提取："Q3毛利率约19%"              ← 四舍五入改变了精确值
LLM提取："Q3毛利率同比下滑"           ← 丢了具体数字

原文："合同违约金为合同总金额的20%"
LLM提取："合同违约金为20%"             ← 丢了关键限定词"合同总金额的"
LLM提取："违约金比例20%"              ← 省略了"合同总金额"这一重要基数

原文："2024年Q3营收同比增长15.3%达4.28亿元"
LLM提取："2024年Q3营收4.28亿元"        ← 丢了增长率信息
LLM提取："Q3营收增长15.3%"            ← 丢了绝对值4.28亿
```

### 核心原则

**数据只锚定，不提取。**

- **value** 必须从原文逐字复制，禁止任何改写
- **summary** 可以由 LLM 生成，但仅用于检索，不作为数据源
- 原文中没有的信息，绝对不能出现在 value 中

### 解决方案：锚定 + 二次验证

#### Step 1：LLM 提取（严格 Prompt）

Prompt 的铁律已在上方"第3级 Prompt"中列出。关键约束：

1. value 必须在原文中逐字出现
2. 禁止计算、换算、四舍五入
3. 不确定则标 uncertain

#### Step 2：机器验证（Go 代码，零 LLM 成本）

```go
// VerifyDataPoint 验证提取的value是否在原文中逐字出现
func VerifyDataPoint(dp DataPoint, sourceQuote string) VerifyResult {
    result := VerifyResult{Passed: true}

    // 1. value必须在original_sentence中逐字出现
    if !strings.Contains(dp.OriginalSentence, dp.Value) {
        result.Passed = false
        result.Reason = fmt.Sprintf("value '%s' 不在 original_sentence 中逐字出现", dp.Value)
    }

    // 2. original_sentence必须在sourceQuote中出现
    if !strings.Contains(sourceQuote, dp.OriginalSentence) {
        result.Passed = false
        result.Reason = "original_sentence 不在原文中出现"
    }

    // 3. 数字一致性检查：value中的数字必须在原文中出现
    valueNums := extractNumbers(dp.Value)
    sourceNums := extractNumbers(sourceQuote)
    for _, vn := range valueNums {
        found := false
        for _, sn := range sourceNums {
            if vn == sn {
                found = true
                break
            }
        }
        if !found {
            result.Passed = false
            result.Reason = fmt.Sprintf("value中的数字 %v 在原文中找不到", vn)
        }
    }

    if !result.Passed {
        dp.Certainty = "uncertain"
        dp.Verification = "pending"
    }

    return result
}
```

#### Step 3：验证结果处理

```
验证通过 (certainty=certain, 机器验证也通过)
  → verification = 'auto'
  → 直接可用，无需人工审核

验证失败 (LLM标certain但机器验证不通过)
  → certainty 降级为 uncertain
  → verification = 'pending'
  → 进入人工审核队列

LLM自评uncertain
  → verification = 'pending'
  → 进入人工审核队列
  → 不管机器验证是否通过，都需要人工确认
```

### 幻觉防护层级

```
第1层：Prompt约束（铁律指令）
  → 阻止大部分幻觉，但无法100%防止

第2层：机器验证（value在原文逐字匹配）
  → 捕获Prompt约束无法防止的数值篡改/换算/推断

第3层：人工审核（uncertain条目）
  → 捕获机器无法判断的歧义情况

第4层：审计日志（所有提取可溯源到原文）
  → 事后发现问题可追溯
```

### 什么情况下 LLM 改写是被允许的？

只有 `analytical` 和 `procedural` 类型允许 LLM 改写：

| 类型 | 能否LLM改写 | 说明 |
|------|-------------|------|
| factual（事实数据） | 绝对不行 | value逐字复制，summary仅用于检索 |
| analytical（分析结论） | 可以 | 但必须附原文证据链 source_quote |
| procedural（流程规范） | 可以 | 但必须附原文证据链 source_quote |

**LLM 生成的 summary 永远只用于检索，不作为数据源。** 数据源只有两个：
1. `data_points[].value`（factual，逐字复制）
2. `source_quote`（原文锚点）
