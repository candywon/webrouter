# 企业知识库 — 开源开发方法律风险分析

> 版本：v1.0 | 日期：2026-05-19

---

## 1. 核心问题

作为开源软件开发方，你提供的是一个**可以采集和处理他人对话内容的工具**。问题在于：

```
使用方（企业）用你的工具采集员工对话 → 违反PIPL
开发方（你）提供了这个工具 + 文档教人怎么采集 → 你有没有责任？
```

---

## 2. 三个法律角色

```
PIPL定义的三个角色：

① 个人信息处理者 — 决定处理目的和方式的组织
   → 使用wr-proxy的企业

② 受托处理者 — 受委托处理个人信息的组织
   → 不适用（你不是受托处理，你提供的是工具）

③ 个人信息 — 被处理的数据主体
   → 企业的员工
```

**关键判定**：开发方既不是"个人信息处理者"（你不决定采集谁的数据），也不是"受托处理者"（你不替任何人处理数据）。你是**工具提供者**。

但"工具提供者"不是法律空白——

---

## 3. 五类风险

### 风险1：帮助侵权（最大风险）

**法律依据**：民法典第1169条 — 教唆、帮助他人实施侵权行为的，应当与行为人承担连带责任。

```
场景：
  你开源了wr-proxy + 写了详细的捕获/提取/存储文档
  某企业拿来用，默认开启capture，不告知员工，不脱敏
  员工起诉企业侵犯隐私权

问题：
  员工能否同时起诉开发方"帮助侵权"？

判定要素：
  1. 开发方是否明知该工具会被用于侵权？
  2. 开发方是否提供了侵权指导？
  3. 开发方是否从侵权中获利？
  4. 开发方是否采取了合理的防护措施？
```

**风险等级：中高**

如果软件默认开启采集、文档鼓励"全量捕获"、没有合规提示，法院可能认定开发方"帮助侵权"。但如果软件默认关闭采集、有合规文档、有技术防护，则很难认定。

**类比案例**：

```
快播案 — 提供视频播放工具+缓存技术
  被认定：明知用户用于侵权仍提供技术支持 → 传播淫秽物品牟利罪

VS

BitTorrent — 提供P2P协议和客户端
  法院认定：工具本身合法，用户侵权行为不由开发者承担

关键区别：
  快播 → 有缓存服务器+主动推荐+从中获利 → "明知+帮助"
  BitTorrent → 纯协议+无中心服务器+不获利 → 工具中立
```

---

### 风险2：产品责任

**法律依据**：民法典第1202-1207条（产品责任）；产品质量法

```
场景：
  wr-proxy的脱敏模块有bug，身份证号没被正确替换
  企业依赖此功能，结果员工身份证泄露
  员工起诉开发方"产品缺陷导致损害"

判定要素：
  1. 软件是否属于"产品"？ → 开源软件一般不被认定为"产品"（无对价）
  2. 是否存在"缺陷"？ → 脱敏bug确实是缺陷
  3. 是否造成损害？ → 身份证泄露
  4. 缺陷与损害是否有因果关系？ → 有
  5. 开源软件免责声明是否有效？ → 通常有效，但有例外
```

**风险等级：低中**

开源软件通常通过License免责，但有两个例外：
1. 开发方明知有缺陷而不修复
2. 开发方提供商业支持/托管服务（有了对价关系）

---

### 风险3：刑法风险（极端情况）

**法律依据**：刑法第253条之一 — 侵犯公民个人信息罪

```
构成要件：
  1. 违反国家有关规定
  2. 向他人出售或者提供公民个人信息
  3. 情节严重（5000条以上/获利5000元以上/造成严重后果）

开发方是否构成？
  ❌ 不出售个人信息（开源软件不涉及数据交易）
  ❌ 不提供个人信息（软件本身不存储/传输任何用户数据）
  ❌ 不"获取"个人信息（开发方不运行软件）

但有一个边界场景：
  如果你提供SaaS托管版 → 你运行了软件 → 你可能成为"个人信息处理者"
  此时如果未合规，可能触及本罪
```

**风险等级：极低（纯开源）** / **中（如果提供托管服务）**

---

### 风险4：开源License合规风险

**法律依据**：开源License合同约束

```
这不是隐私问题，但容易被忽略：

1. wr-proxy用了哪些开源依赖？
   Go的依赖：gin/echo/gorm/sqlite-driver等
   各自有MIT/Apache/BSD等License
   → 需要在开源时声明依赖和License

2. 你选择什么License开源？
   MIT → 最宽松，别人可以闭源商业使用
   Apache 2.0 → 宽松+专利授权+声明变更
   GPL v3 → 衍生作品必须同样开源（传染性）
   AGPL v3 → 网络服务也算分发（最强传染性）

3. 如果用了GPL依赖 → 你的代码必须也用GPL
   如果没注意 → 违反GPL → 被依赖方起诉
```

**风险等级：中**（License选错或依赖声明缺失很常见）

---

### 风险5：跨境数据传输（如果海外用户）

**法律依据**：PIPL第38-43条（跨境提供个人信息）

```
场景：
  开源软件被海外企业使用
  海外企业用wr-proxy采集中国员工对话
  数据存储在海外服务器

问题：
  这构成"个人信息跨境传输"
  谁负责合规？→ 使用方（企业），不是开发方

但：
  如果你在海外提供托管服务/SaaS → 你就是跨境传输方
  → 需要通过安全评估/标准合同/认证
```

**风险等级：无（纯开源）** / **高（如果提供海外SaaS）**

---

## 4. 风险总览

| 风险 | 纯开源 | 开源+商业支持 | 开源+SaaS托管 |
|------|--------|-------------|--------------|
| 帮助侵权 | 中高 | 高 | 高 |
| 产品责任 | 低中 | 中 | 高 |
| 刑法(侵犯个人信息) | 极低 | 低 | 中 |
| License合规 | 中 | 中 | 中 |
| 跨境传输 | 无 | 低 | 高 |

**核心结论**：纯开源模式风险可控，关键是做好"工具中立性"防护。一旦提供SaaS托管，风险大幅上升。

## 5. 防护措施

### 措施1：默认安全（最关键）

```
法律逻辑：
  如果你的软件默认开启采集 → 法院可能认为你在"鼓励侵权"
  如果你的软件默认关闭采集 → 法院认定"工具中立"

必须做到：
  ✓ knowledge_capture_enabled 默认 = 0（已是）
  ✓ rag_enabled 默认 = 0（已是）
  ✓ 首次启动必须手动 --knowledge-capture 才能启用（已是）
  ✓ 脱敏规则默认开启（已是）
  ✓ 同意检查(consent_required)默认开启（需新增）

绝对不能做：
  ✗ 默认采集所有对话
  ✗ 提供一键"全量采集"按钮
  ✗ 配置文件中capture=on作为示例
```

### 措施2：强免责声明（License + README）

**开源License选择建议**：Apache 2.0

```
原因：
  1. 明确的专利授权条款
  2. 明确的免责条款（Section 7/8）
  3. 要求保留NOTICE文件（可放入合规提示）
  4. 不传染（使用者可以闭源商业使用）
  5. 行业主流（Kubernetes/Docker/TensorFlow都用Apache 2.0）
```

**NOTICE文件**（Apache 2.0要求保留）：


```markdown
the proxy. **All capture features are DISABLED BY DEFAULT.**

1. Inform all users that their conversations may be captured
```

### 措施3：技术防护层

```
第1层：默认关闭
  所有采集功能默认=0，必须手动配置开启

第2层：启用时强制提示
  --knowledge-capture 启动时打印：
  ⚠️  WARNING: Knowledge capture is ENABLED.
  ⚠️  You MUST obtain consent from all users before capturing.
  ⚠️  You MUST comply with applicable privacy laws.
  ⚠️  See docs/knowledge-base/法律合规分析.md for guidance.

第3层：运行时检查
  如果capture开启但consent_given未配置 → 打印警告
  如果脱敏模块被禁用 → 打印警告
  如果raw表存储超30天未清理 → 打印警告

第4层：代码注释
  在DeliverKnowledge函数入口注释：
  // PRIVACY: This function captures user conversations.
  // MUST only be called when the user has given informed consent.
  // See 法律合规分析.md for legal requirements.
```

### 措施4：文档中嵌入合规要求

```
所有涉及数据采集的文档中，必须包含合规提示：

技术方案文档 → 开头加：
  ⚠️ 本文档描述的技术方案涉及对话内容采集。
  实施前必须完成隐私合规评估。详见法律合规分析.md。

捕获层设计 → shouldCapture函数文档：
  ⚠️ 信号筛选不替代合规审查。即使通过筛选的对话，
  也可能包含个人信息。脱敏和同意机制不可跳过。

API设计 → 每个可能返回个人数据的API：
  ⚠️ 此API可能返回包含个人信息的知识条目。
  调用方应确保检索结果的使用符合隐私法规。
```

### 措施5：明确使用条款（TOU）

即使在开源License之外，也应在项目根目录放置 USE_TERMS.md：

```markdown
# Terms of Use

## Privacy Obligations

By using the knowledge capture features of this software, 
you agree to:

1. Obtain informed consent from all individuals whose 
   conversations may be captured, before enabling capture
2. Not disable or bypass the built-in desensitization module
3. Provide data access and deletion rights to all individuals
4. Not use captured data for purposes other than those disclosed 
   to the individuals (e.g., not for employee surveillance)
5. Comply with all applicable privacy and data protection laws

## Prohibited Uses

This software must NOT be used to:
- Covertly monitor or surveil individuals without their knowledge
- Collect personal information without proper legal basis
- Process sensitive personal information (health, biometric, 
  financial, minor's data) without required additional safeguards
- Use captured personal data for purposes not disclosed to 
  the data subjects

## Liability

The operators of this software bear sole responsibility for 
compliance with applicable laws. The software developers are 
not liable for any violation of privacy or data protection 
laws committed by operators.
```

### 措施6：不提供"一键侵权"功能

```
有些功能看似方便，实际上是法律陷阱：

❌ 不要做：
  - "一键全量捕获"按钮
  - "跳过同意检查"配置项
  - "禁用脱敏"配置项
  - "导出原始对话"API（导出脱敏摘要可以）
  - 配置示例中capture=on

✅ 可以做：
  - 需要手动修改配置文件+命令行参数才能开启
  - 同意检查不可跳过（除非改源码，但那是使用者的选择）
  - 脱敏模块不可通过配置关闭（需改源码）
  - 导出的是知识条目不是原始对话
```

## 6. 与同类开源项目的对比

| 项目 | 有没有采集能力 | 合规防护 | License |
|------|--------------|---------|---------|
| LiteLLM | 有（logging） | 默认关闭 | MIT |
| One-API | 有（日志记录） | 默认关闭 | MIT |
| OpenRouter | 有（API日志） | 商业服务有合规 | 商业 |
| dify | 有（对话记录） | 有用户同意机制 | Apache 2.0 |
| langchain | 有（记忆/检索） | 无特殊合规 | MIT |
| **wr-proxy** | **有（捕获+提取+存储）** | **默认关闭+脱敏+同意** | **建议Apache 2.0** |

**wr-proxy的采集能力比同类项目更深**（不只是记录日志，还提取知识+存储+共享），所以合规防护也必须比它们更强。

## 7. 风险场景与应对

### 场景1：某企业用你的软件隐蔽监控员工

```
企业行为：capture默认开启，不告知员工
员工发现后起诉企业，同时追索开发方

开发方抗辩：
  1. 软件默认关闭采集 → 企业主动修改了配置
  2. 启动时有WARNING提示 → 企业看到了合规要求
  3. README/USE_TERMS明确写了禁止隐蔽监控
  4. 开发方未从企业使用中获利
  5. 开发方无法控制企业如何部署

判定倾向：开发方不承担责任
  前提：上述防护措施都已到位
```

### 场景2：脱敏模块有bug，身份证泄露

```
企业依赖脱敏功能，但某格式身份证未被正确替换
员工起诉企业和开发方

开发方抗辩：
  1. 开源软件免责声明（Apache 2.0 Section 7/8）
  2. 软件无保证（AS IS），使用者应自行验证
  3. 文档中标注了脱敏覆盖率，未承诺100%
  4. 发现bug后及时修复

风险：
  如果bug存在很久且有人报过但你没修 → 可能被认定"明知缺陷"
  如果及时修复 → 基本无责
```

### 场景3：有人在社区发帖问"怎么绕过同意检查"

```
社区用户："我想默认开启capture不想做同意流程，怎么改代码？"

你的回应：
  ❌ 不能回答："改这里这里就行了" → 构成"帮助侵权指导"
  ✅ 应该回答："我们不建议跳过同意检查，这违反PIPL。
     如果你有合法场景，请参考法律合规分析.md。"

社区管理：
  在Issue/讨论区置顶合规提示
  对明显侵权的求助帖进行提醒
  不需要主动删除（那反而显得你在监控社区）
```

## 8. 开源前检查清单

```
□ License选择：Apache 2.0（推荐）/ MIT
□ NOTICE文件：包含合规警告
□ README.md：隐私合规提示在显著位置
□ USE_TERMS.md：使用条款，明确禁止隐蔽监控
□ 所有采集功能默认关闭
□ 启动时WARNING提示
□ 脱敏模块不可通过配置关闭
□ 同意检查机制（consent_required）
□ 法律合规分析文档随项目发布
□ 代码注释中标注PRIVACY关键点
□ 依赖License声明（go.sum + NOTICE）
□ 无"一键全量采集"功能
□ 无"跳过同意"配置项
□ 无"禁用脱敏"配置项
□ 导出功能只导出知识条目不导出原始对话
□ 配置示例中capture=off
```

## 9. 核心结论

```
开发方的风险 ≠ 使用方的风险

使用方的风险：直接违法（采集了个人信息没合规）
开发方的风险：间接责任（提供了侵权工具+指导）

降低开发方风险的核心原则：

1. 工具中立 — 默认关闭采集，使用者主动开启
2. 不可跳过 — 合规检查不能通过配置关闭（只能改源码）
3. 充分告知 — 文档/启动/代码处处提醒合规义务
4. 不提供指导 — 不教人怎么绕过合规
5. 及时修bug — 脱敏等安全相关缺陷要快速响应
6. 明确免责 — License+NOTICE+USE_TERMS多层免责

做好了这些，开发方的法律风险极低。
做不好（默认开启+无提示+鼓励全量采集），风险中高。
```
