# 企业知识库 — 数据零幻觉方案

> 版本：v1.0 | 日期：2026-05-14

---

## 1. 问题定义

LLM 提炼事实数据时会产生三类幻觉：

### 类型1：换算幻觉

```
原文："Q3毛利率同比收窄3.2个百分点至18.7%"
LLM提取："Q3毛利率下降3.2%"       ← 错误！收窄3.2个百分点≠下降3.2%
LLM提取："Q3毛利率为3.2%"         ← 严重错误！把百分点当绝对值
```

### 类型2：省略幻觉

```
原文："合同违约金为合同总金额的20%"
LLM提取："合同违约金为20%"         ← 丢了关键限定词"合同总金额的"
```

### 类型3：近似幻觉

```
原文："2024年Q3营收同比增长15.3%达4.28亿元"
LLM提取："Q3营收增长15%"           ← 四舍五入改变了精确值
LLM提取："Q3营收约4.3亿元"         ← 近似改变了精确值
```

## 2. 核心原则

```
数据只锚定，不提取。

- value 必须从原文逐字复制，禁止任何改写
- summary 可以由LLM生成，但仅用于检索，不作为数据源
- 原文中没有的信息，绝对不能出现在value中
```

## 3. 四层防护体系

### 第1层：Prompt约束

```
你是企业数据提取引擎。从以下对话原文中提取事实数据。

【铁律】
1. value字段必须从原文逐字复制，禁止任何改写、换算、推断
2. 如果原文说"收窄3.2个百分点至18.7%"，value只能写"18.7%"
3. 如果数据不确定或有歧义，certainty标为uncertain
4. original_sentence必须包含该数据的完整原句
5. 禁止计算、换算、四舍五入、补零
6. 原文中没有的数据，绝对不能出现在value中

对话原文：
---
{conversation}
---

输出JSON：
{
  "data_points": [
    {
      "metric": "指标名称",
      "value": "从原文逐字复制的值",
      "period": "时间范围",
      "unit": "单位",
      "original_sentence": "包含此数据的原文完整句子",
      "certainty": "certain 或 uncertain"
    }
  ],
  "has_no_data": false
}
```

防护能力：阻止大部分幻觉（约80-90%），但LLM仍可能"自作聪明"地换算。

### 第2层：机器验证（Go代码，零LLM成本）

```go
// VerifyDataPoint 验证提取的data_point是否在原文中逐字出现
func VerifyDataPoint(dp DataPoint, sourceQuote string) VerifyResult {
    result := VerifyResult{Passed: true}

    // 规则1：value必须在original_sentence中逐字出现
    if !strings.Contains(dp.OriginalSentence, dp.Value) {
        result.Passed = false
        result.Reason = fmt.Sprintf(
            "value '%s' 不在 original_sentence 中逐字出现", dp.Value)
    }

    // 规则2：original_sentence必须在sourceQuote中出现
    if !strings.Contains(sourceQuote, dp.OriginalSentence) {
        result.Passed = false
        result.Reason = "original_sentence 不在原文中出现"
    }

    // 规则3：数字一致性检查
    // value中出现的所有数字必须在原文的对应位置也出现
    valueNums := extractNumbers(dp.Value)
    sourceNums := extractNumbers(sourceQuote)
    for _, vn := range valueNums {
        found := false
        for _, sn := range sourceNums {
            if vn == sn {
                found = true
                break
            }
        }
        if !found {
            result.Passed = false
            result.Reason = fmt.Sprintf(
                "value中的数字 %v 在原文中找不到", vn)
        }
    }

    return result
}

// extractNumbers 从文本中提取所有数字
func extractNumbers(text string) []string {
    re := regexp.MustCompile(`\d+\.?\d*`)
    return re.FindAllString(text, -1)
}
```

防护能力：捕获Prompt无法防止的数值篡改/换算/推断（约95-99%）。

### 第3层：人工审核

验证结果处理逻辑：

```
LLM自评certainty=certain + 机器验证通过
  → verification = 'auto'
  → 直接可用

LLM自评certainty=certain + 机器验证不通过
  → certainty降级为uncertain
  → verification = 'pending'
  → 进入人工审核队列

LLM自评certainty=uncertain（不管机器验证结果）
  → verification = 'pending'
  → 进入人工审核队列
```

防护能力：捕获机器无法判断的歧义情况（100%）。

### 第4层：审计溯源

每条知识都保留完整的溯源链：

```
wr_knowledge_items.source_request_id  → 原始请求ID
wr_knowledge_items.source_turn_index  → 第几轮对话
wr_knowledge_items.source_quote       → 原文片段（精确引用）
wr_knowledge_items.source_char_start  → 原文中的字符起始位置
wr_knowledge_items.source_char_end    → 原文中的字符结束位置
```

任何知识都可以追溯到原始对话的精确位置，事后发现问题可追溯。

## 4. 验证流程图

```
LLM提取data_points
       │
       ▼
  机器验证每个data_point
       │
       ├── value在original_sentence中逐字出现？
       │     ├── 是 → 继续
       │     └── 否 → certainty降为uncertain
       │
       ├── original_sentence在source_quote中出现？
       │     ├── 是 → 继续
       │     └── 否 → certainty降为uncertain
       │
       └── value中的数字都在原文中出现？
             ├── 是 → 继续
             └── 否 → certainty降为uncertain

  最终判断：
       │
       ├── 所有验证通过 + LLM自评certain
       │     → verification='auto'，直接入库
       │
       └── 任一验证失败 或 LLM自评uncertain
             → verification='pending'，入审核队列
```

## 5. LLM改写权限矩阵

| 知识类型 | value | summary | source_quote | 说明 |
|----------|-------|---------|--------------|------|
| factual | 逐字复制，不可改写 | LLM可生成，仅用于检索 | 精确引用 | 数据零幻觉 |
| analytical | 无data_points | LLM可提炼 | 必须附原文证据 | 结论需溯源 |
| procedural | 无data_points | LLM可提炼 | 必须附原文证据 | 流程需溯源 |

**LLM生成的summary永远不作为数据源。** 数据源只有：
1. `data_points[].value`（factual，逐字复制）
2. `source_quote`（原文锚点）

## 6. 测试用例

### 测试1：换算幻觉应被拦截

```
原文："Q3毛利率同比收窄3.2个百分点至18.7%"
LLM提取：value="3.2%"（错误，应为18.7%）

机器验证：strings.Contains("Q3毛利率同比收窄3.2个百分点至18.7%", "3.2%")
→ true！因为"3.2个"中确实包含"3.2"

问题：纯字符串匹配不够，需要更精确的验证。
```

**改进：数值上下文验证**

```go
// 不仅验证数字出现，还验证数字的上下文
func verifyNumericContext(value, source string) bool {
    // 提取value中的数字
    numRegex := regexp.MustCompile(`(\d+\.?\d*)\s*(%|元|万|亿|个|天|人|件)?`)
    matches := numRegex.FindAllStringSubmatch(value, -1)

    for _, match := range matches {
        num := match[1]
        unit := ""
        if len(match) > 2 {
            unit = match[2]
        }

        // 在原文中找这个数字+单位的组合
        pattern := regexp.MustCompile(num + `\s*` + unit)
        if !pattern.MatchString(source) {
            return false
        }
    }
    return true
}
```

### 测试2：省略幻觉应被拦截

```
原文："合同违约金为合同总金额的20%"
LLM提取：value="20%"（省略了"合同总金额的"）

机器验证：value="20%" 在原文中确实出现
→ 但这不算幻觉，因为20%本身就是准确的值
→ 省略的是限定条件，不是数值本身

处理：metric字段应写"合同违约金（占合同总金额）"来保留限定信息
→ 这需要Prompt约束来保证
```

### 测试3：近似幻觉应被拦截

```
原文："营收同比增长15.3%达4.28亿元"
LLM提取：value="15%"（四舍五入）

机器验证：strings.Contains("...15.3%...", "15%")
→ false！"15%"在原文中找不到（原文是15.3%）
→ 拦截成功！certainty降为uncertain
```

## 7. 持续改进

验证规则需要根据实际运行数据迭代：

1. **运行1周**：统计机器验证拦截率，如果<5%说明LLM Prompt已经足够好
2. **分析uncertain条目**：分类幻觉类型，针对性加强Prompt或验证规则
3. **每月Review**：审核队列的通过率/拒绝率，调整验证阈值

目标：auto验证率 > 90%，pending需人工审核 < 10%。
