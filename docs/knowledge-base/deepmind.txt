    核心难点拆解

    你提出的是三个递进的问题：


    1. 怎么识别"这是知识"而不是"这是闲聊"？
    2. 怎么按部门归类，且分类体系能跟着业务演化？
    3. 怎么保证数据类知识绝对准确，零幻觉？


    逐一拆开。



    难点1：知识识别 — 不能靠关键词

    关键词方案的致命问题：


    "帮我查一下合同到期时间" → 关键词"合同"命中 → 但这只是查询，不是知识
    "这份合同的违约金条款和行业惯例有冲突" → 同样命中"合同" → 这才是知识

    "Q3毛利率3.2%" → 关键词啥也匹配不到 → 但这是高价值数据知识
    "分析一下毛利率" → 关键词匹配到了 → 但这只是个问题，不是知识


    关键词匹配的是"话题"，不是"知识密度"。真正的分界线是：

    知识 = 包含确定性的信息产出（结论、数据、规则、判断）

    我的方案是多级漏斗：


    第1级：信号筛选（零成本，纯规则）
    ├── 丢弃：单轮短问答（<3轮且响应<200字）
    ├── 丢弃：纯编程/翻译/改写类（检测code block占比>60%）
    ├── 丢弃：明确的无知识意图（"帮我写个函数""翻译成英文"）
    └── 通过：进入第2级

    第2级：知识密度评估（低成本LLM，批量处理）
    ├── 输入：对话全文
    ├── 输出：{ has_knowledge: bool, knowledge_type: "data|conclusion|rule|none", confidence: 0-1 }
    ├── 每批20条对话，一次API调用
    ├── 只用最便宜模型（qwen3-coder-flash）
    └── has_knowledge=false 的直接丢弃

    第3级：知识提取（中成本LLM，逐条处理）
    ├── 只处理第2级通过的条目
    ├── 严格提取 + 原文锚定（下面详述）
    └── 输出带置信度的结构化知识


    关键是第1级要尽可能粗筛掉噪音——实际企业场景中，80%以上的API调用是编程、翻译、改写，真正产生业务知识的可能不到20%。



    难点2：部门归类 — 不能先定义再归类

    传统做法是"先建分类树，再把知识挂上去"。问题是你永远建不完：


    年初：销售部、售后部、财务部
    年中：新增跨境电商组（归销售还是独立？）
    年末：跨境电商变成事业部，下面又有自己的法务和财务


    正确做法是两层归类，只有一层是确定的：


    第1层：部门归属 → 由Token确定（100%准确）
    ├── Token本来就是按部门/员工分配的
    ├── Token "张三-财务部" 产生的知识 → 归属财务部
    ├── 不需要LLM猜，天然准确
    └── 这是确定性维度，不会错

    第2层：业务域 → 由LLM提取（需验证，可演化）
    ├── 初始预设：legal/finance/hr/admin/sales/service/tech
    ├── LLM提取时允许返回 "new_domain" 标记
    ├── 发现新域 → 入库但标记 pending，需管理员确认
    └── 确认后自动加入可选域列表


    业务域演化的具体机制：

    sql
    wr_knowledge_domains 表
    ├── id, domain_code, domain_name, department
    ├── status: "active" | "pending" | "merged"
    ├── sample_count  -- 该域下的知识条数
    ├── auto_keywords -- LLM自动提炼的关键词
    ├── created_at, confirmed_at, confirmed_by
    └── merged_into   -- 如果被合并到其他域

    -- 当 pending 域积累超过20条知识时，
    -- 管理后台弹出提示："发现新业务域'跨境电商'，是否确认？"
    -- 管理员可以：确认 / 合并到已有域 / 忽略


    这样分类体系是从数据中生长出来的，不是人为预设的。



    难点3：数据类知识零幻觉 — 最关键

    这是最核心的问题。LLM提炼知识的最大风险：


    原文："Q3毛利率同比收窄3.2个百分点至18.7%"
    LLM提取："Q3毛利率下降3.2%" ← 错误！收窄3.2个百分点≠下降3.2%
    LLM提取："Q3毛利率为3.2%" ← 严重错误！把百分点当成了绝对值

    原文："合同违约金为合同总金额的20%"
    LLM提取："合同违约金为20%" ← 丢了关键限定词"合同总金额的"


    解决方案：数据类知识不提取，只锚定。

    核心原则：


    事实/数据类知识 → 只存原文片段 + 位置锚点，LLM不做任何改写
    结论/规则类知识 → LLM可以提炼，但必须附原文证据链


    具体设计：

    sql
    wr_knowledge_items 表
    ├── id
    ├── type: "factual" | "analytical" | "procedural"
    │   ├── factual: 事实数据（金额、比例、日期、数量）
    │   ├── analytical: 分析结论（原因分析、趋势判断）
    │   └── procedural: 流程规范（审批流程、操作步骤）
    │
    ├── -- 提取内容 --
    ├── title                    -- LLM生成的标题（仅用于展示/检索）
    ├── summary                  -- LLM生成的摘要（仅用于检索，不作为数据源）
    ├── domain_code              -- 业务域
    ├── department               -- 部门（来自Token）
    │
    ├── -- 关键：原文锚定 --
    ├── source_request_id        -- 原始请求ID
    ├── source_turn_index        -- 第几轮对话
    ├── source_quote             -- 原文片段（精确引用，不改动一字）
    ├── source_char_range        -- 原文中的字符位置 [start, end]
    │
    ├── -- 数据类专门字段 --
    ├── data_points JSON         -- 只用于factual类型
    │   [
    │     {
    │       "metric": "毛利率",
    │       "value": "18.7%",           -- 从原文精确复制，不做任何计算
    │       "period": "Q3",
    │       "original_sentence": "Q3毛利率同比收窄3.2个百分点至18.7%",
    │       "certainty": "certain"      -- certain/uncertain
    │     }
    │   ]
    │
    ├── -- 验证 --
    ├── confidence REAL           -- 0.0-1.0，LLM自评
    ├── verification: "auto" | "pending" | "verified" | "rejected"
    ├── verified_by INTEGER       -- 审核人
    ├── verified_at DATETIME
    │
    ├── -- 元数据 --
    ├── token_id, token_name
    ├── model_name
    ├── sensitivity: "low" | "medium" | "high" | "restricted"
    ├── retention_until DATETIME  -- 保留期限
    ├── created_at, updated_at


    数据类知识的提取Prompt（严格模式）：


    你是企业数据提取引擎。从以下对话原文中提取事实数据。

    【铁律】
    1. value字段必须从原文逐字复制，禁止任何改写、换算、推断
    2. 如果原文说"收窄3.2个百分点至18.7%"，value只能写"18.7%"，不能写"下降3.2%"
    3. 如果数据不确定或有歧义，certainty标为uncertain
    4. original_sentence必须包含该数据的完整原句
    5. 禁止计算、换算、四舍五入、补零
    6. 原文中没有的数据，绝对不能出现在value中

    对话原文：

    {conversation}


    输出JSON：
    {
      "data_points": [
        {
          "metric": "指标名称",
          "value": "从原文逐字复制的值",
          "period": "时间范围（如有）",
          "unit": "单位（如有）",
          "original_sentence": "包含此数据的原文完整句子",
          "certainty": "certain 或 uncertain"
        }
      ],
      "has_no_data": false
    }


    然后做二次机器验证：

    go
    // 提取完成后，自动验证value是否原文中出现
    func VerifyDataPoints(dp DataPoint, sourceQuote string) bool {
        // value必须在original_sentence中逐字出现
        if !strings.Contains(dp.OriginalSentence, dp.Value) {
            dp.Certainty = "uncertain"
            dp.Verification = "pending"  // 需人工审核
            return false
        }
        return true
    }


    这一步是关键——机器可验证。不管LLM怎么提，只要提取的value在原文中找不到精确匹配，就标记为uncertain，必须人工审核。

    总结新增的超长上下文分析设计要点：

    核心思路：把一个业务域的全部知识（而不是几条）灌入大模型做全局分析

    模式对比：
    - knowledge_search：查几条 → "Q3毛利率多少？"
    - knowledge_analyze：灌全域 → "财务域所有数据有什么矛盾和趋势？"

    关键技术决策：

    1. 模型自动选择 — 按知识量自动路由：
       - <8万token → DeepSeek-chat（128K窗口，最便宜）
       - 8-20万 → Gemini 2.0 Flash（1M窗口，0.005/M）
       - 20-80万 → Gemini 2.5 Pro（1M窗口，高质量）
       - >80万 → Qwen-Long（10M窗口）

    2. 分段汇总策略 — 单域超出模型窗口时，按时间段分段独立分析，再汇总

    3. 复用wr-proxy路由 — 分析请求不走直连，走wr-proxy自身的代理流程，享受智能路由+重试+降级

    4. 分析结果不入知识库 — 分析是大模型推理输出，不是锚定原文的事实，想入库需走正常提取+验证流程

    5. MCP工具 knowledge_analyze — 智能体对话中说"帮我分析下财务域"自动触发

    6. 成本极低 — 单域分析 < 1分钱，全域分析 < 1毛钱



    MCP之外，增强智能体专业知识/记忆还有5种方式：

    ① System Prompt预注入 — 最简单最实用
    - wr-proxy在转发请求前，自动在messages最前面插入一条system消息，包含公司简介、部门职能、关键流程提示
    - 零额外API调用、零延迟、用户无感知
    - 工作量0.5天，可以立刻做

    ② RAG自动注入 — 常见问题自动增强
    - 用户提问时，wr-proxy自动对prompt做语义检索，把高相关度的知识片段注入system prompt
    - 和MCP的区别：RAG是"自动的"，AI不需要决定调不调；MCP是"AI主动调"
    - 80%的常见问题RAG就能覆盖，剩下20%深度查询走MCP

    ③ 持久记忆层 — 跨会话记住用户信息
    - "我在负责跨境电商项目" → 提取为记忆 → 下次新会话AI自动知道
    - 和知识库的区别：知识库是公司级共享，记忆是用户级私有
    - 解决"每次新对话都要重新介绍自己"的痛点

    ④ 对话摘要压缩 — 长对话不丢失
    - 对话超20轮后，早期内容压缩为2K tokens的摘要
    - 配合知识库：压缩时同时把有价值片段投递到知识库

    ⑤ 知识微调 — 远期方向，当前不建议
    - 用公司数据微调模型让知识内化
    - 成本高、更新难，RAG+MCP已能解决90%问题

    推荐组合：
    - 一期：System Prompt + RAG（覆盖80%场景，工作量2.5天）
    - 二期：+ MCP + 持久记忆（覆盖95%场景）
    - 三期：+ 对话摘要 + 全域分析（全覆盖）
