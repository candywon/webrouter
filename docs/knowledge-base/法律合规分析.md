# 企业知识库 — 法律合规分析

> 版本：v1.0 | 日期：2026-05-19

---

## 1. 问题本质

企业知识库之所以面临法律合规问题，根本原因是**数据主体和数据控制者分离**：

```
AI工具模式（Hermes/Claude Code/OpenClaw）：
  用户 ←→ AI工具 ←→ 大模型API
  数据主体 = 数据控制者 = 同一个人 → 无第三方采集问题

企业知识库模式（wr-proxy）：
  员工 ←→ wr-proxy ←→ 大模型API
              │
              └── 企业在这里采集、存储、提取、共享
              数据主体(员工) ≠ 数据控制者(企业) → 受法律约束
```

**类比**：AI工具是"私人笔记本"，企业知识库是"公司档案柜"。私人笔记本没人管你记什么，公司档案柜装的是别人的信息，法律就要管了。

## 2. 涉及的法律框架

### 2.1 民法典（第1032-1039条）

| 条款 | 内容 | 约束 |
|------|------|------|
| 第1032条 | 自然人享有隐私权 | 任何组织不得以刺探、侵扰、泄露、公开等方式侵害隐私权 |
| 第1033条 | 隐私权侵害行为 | 不得拍摄、窥视、窃听、公开他人的私密活动 |
| 第1034条 | 个人信息受法律保护 | 个人信息中的私密信息，适用隐私权规定 |
| 第1035条 | 个人信息处理原则 | 合法、正当、必要原则；明示处理目的、方式、范围 |
| 第1038条 | 信息安全义务 | 不得泄露或篡改其收集、存储的个人信息 |

### 2.2 个人信息保护法（PIPL）

| 条款 | 内容 | 约束 |
|------|------|------|
| 第4条 | 处理个人信息需遵循合法、正当、必要原则 | — |
| 第5条 | 不得过度收集个人信息 | 最小必要原则 |
| 第7条 | 处理个人信息应取得个人同意 | 知情同意原则 |
| 第13条 | 处理个人信息应当取得个人的同意 | 明示同意 |
| 第14条 | 同意应当由个人在充分知情的前提下自愿、明确作出 | 不能默认推定 |
| 第24条 | 个人有权撤回同意 | 可退出机制 |
| 第28条 | 敏感个人信息需单独同意 | 身份证/健康/金融/生物识别等 |
| 第47条 | 个人有权请求删除个人信息 | 数据删除权 |
| 第55条 | 处理敏感个人信息应事前影响评估 | 隐私影响评估 |

### 2.3 数据安全法

| 条款 | 内容 | 约束 |
|------|------|------|
| 第27条 | 开展数据处理活动应依法建立数据安全管理制度 | 制度建设 |
| 第29条 | 开展数据处理活动应加强风险监测 | 安全监测 |
| 第30条 | 重要数据处理者应定期开展风险评估 | 定期评估 |

## 3. 风险分析

### 3.1 高风险场景

| 场景 | 风险等级 | 触犯法律 | 后果 |
|------|---------|---------|------|
| 隐蔽采集：员工不知道对话被捕获 | **极高** | PIPL第7/13条 | 责令改正+警告+罚款 |
| 未脱敏存储原始对话 | **高** | 民法典1035条/PIPL第5条 | 民事赔偿+行政处罚 |
| 捕获的对话含敏感个人信息 | **高** | PIPL第28条 | 罚款+可能刑事责任 |
| 员工要求删除但无法删除 | **高** | PIPL第47条 | 责令改正+罚款 |
| 知识被跨部门共享（含个人信息） | **中高** | 民法典1038条 | 民事赔偿 |
| 捕获数据用于绩效考核等未告知目的 | **中** | PIPL第6条（目的限制） | 行政处罚 |

### 3.2 合规与违法的边界

**合规场景**：
```
1. 公司明确告知 + 员工明确同意
   员工签署知情同意书，了解对话会被提取业务知识

2. Token级开关 — 默认关闭
   knowledge_capture_enabled 默认 = 0
   员工主动开启 → 构成"明示同意"

3. 脱敏在捕获之前
   存入知识库的已是脱敏内容，不包含可识别个人的信息

4. 只提取业务知识
   "Q3毛利率18.7%" — 不涉及个人
   不提取"张三工资3万" — 涉及个人信息
```

**违法场景**：
```
1. 隐蔽采集 — knowledge_capture_enabled默认=1，不告知
2. 未脱敏存储 — 原始对话完整保存含身份证/手机号
3. 滥用数据 — 告知说"用于知识库"实际用于绩效评估
4. 无删除机制 — 员工要求删除但无法操作
5. 敏感信息未特殊处理 — 健康信息/金融信息混在普通知识中
```

## 4. 为什么Claude Code/Hermes/OpenClaw没有这个问题

### 4.1 核心区别

| 维度 | AI工具 | 企业知识库 |
|------|--------|-----------|
| 数据流向 | 用户→AI→用户（闭环） | 员工→网关→知识库→其他人（开环） |
| 谁控制数据 | 用户自己 | 企业（管理员） |
| 数据用途 | 当次推理 | 持久化+提取+共享 |
| 有没有第三方 | 没有 | 有（企业） |
| 数据汇聚 | 个人隔离 | 多员工统一存储 |
| 可见范围 | 只有自己 | 同部门/跨部门 |
| 法律定性 | 个人使用工具 | 组织处理个人信息 |

### 4.2 逐工具分析

**Claude Code**：
- 开发者本地运行 → 调Anthropic API → 返回结果
- Anthropic是"受托处理者"，不使用API数据训练模型
- 开发者自己决定发什么、存什么，没有第三方采集

**Hermes Agent**：
- memory功能是"个人笔记"性质，存在用户本地 ~/.hermes/
- 用户自己完全控制：可查看、删除、关闭
- 不存在跨用户数据汇聚和共享

**OpenClaw**：
- 与Claude Code类似，用户=控制者
- API调用是受托处理，不是主动收集

### 4.3 它们也不是完全没有风险

```
Hermes — 如果企业IT管理 ~/.hermes/ 目录：
  等于企业查看员工记忆 → 同企业知识库的法律问题

Claude Code — 如果企业统一管理API Key：
  企业通过API后台看到代码和对话 → 也有隐私问题

ChatGPT企业版 — OpenAI已遇到此问题：
  明确承诺"不使用企业数据训练模型" → 正是隐私顾虑的回应
```

**结论**：不是这些工具没有隐私问题，而是当前使用方式碰巧没触发。企业一旦系统化利用员工对话数据，问题完全一样。

### 4.4 类比：电话录音

```
个人用手机录音 → 自己听，没问题
公司给员工配电话并自动录音 → 必须告知+同意+有制度
公司把录音提取为"客户信息库" → 更严格的法律要求

AI工具 = 个人手机
企业知识库 = 公司电话录音系统
```

## 5. wr-proxy现有设计的合规评估

### 5.1 已有的合规设计（✅）

| 设计点 | 满足的法律要求 | 说明 |
|--------|--------------|------|
| Token级开关，默认关闭 | PIPL第13条（明示同意） | 员工主动开启才捕获 |
| 脱敏在捕获之前 | PIPL第5条（最小必要） | 手机号/身份证等已被替换 |
| knowledge_department由Token关联 | 数据分类 | 部门是岗位信息不是隐私 |
| retention_until保留期限 | PIPL第19条（存储期限） | 过期自动清理 |
| 领域风险分级 | PIPL第28条（敏感信息特殊处理） | legal/finance域只注入verified知识 |
| MCP工具只读 | 最小授权 | 智能体不能修改/删除知识 |
| 部门权限隔离 | PIPL第51条（访问控制） | 不同部门看不到对方敏感知识 |

### 5.2 缺失的合规设计（❌）

| 缺失项 | 违反的法律 | 风险等级 | 改造优先级 |
|--------|----------|---------|-----------|
| **知情同意流程** | PIPL第7/13/14条 | 极高 | P0 |
| **员工数据删除入口** | PIPL第47条 | 高 | P0 |
| **raw表数据最小化** | PIPL第5条（最小必要） | 高 | P0 |
| **审计日志** | 数据安全法第27条 | 中 | P1 |
| **敏感信息二次检测** | PIPL第28条 | 中高 | P1 |
| **隐私影响评估** | PIPL第55条 | 中 | P2 |

## 6. 合规改造方案

### 改造1：知情同意机制（P0，1天）

**法律依据**：PIPL第7条（知情同意）、第13条（取得个人同意）、第14条（充分知情+自愿明确）

**方案**：首次使用弹窗确认

```
员工第一次通过Token调用API时，wr-proxy拦截：

请求 → 检查Token.consent_given
  ├── consent_given=1 → 正常代理
  └── consent_given=0 → 返回：
      {
        "error": {
          "type": "consent_required",
          "message": "使用企业AI助手需同意知识捕获协议",
          "consent_url": "https://wr.company.com/consent?token=xxx"
        }
      }

员工打开链接 → 阅读3项核心内容 → 勾选同意 → consent_given=1 → 以后不再拦截
```

**同意书核心内容**（必须包含）：

```
1. 【收集什么】我们会自动提取对话中的业务知识（数据、结论、流程）
2. 【不收集什么】不会提取个人隐私信息（已自动脱敏处理）
3. 【怎么用】提取的知识存储在企业知识库，供同部门同事检索参考
4. 【你的权利】你可以随时关闭捕获功能、查看自己的数据、要求删除

☐ 我已阅读并同意以上内容
```

**DB扩展**：

```sql
ALTER TABLE wr_tokens ADD COLUMN consent_given INTEGER DEFAULT 0;
ALTER TABLE wr_tokens ADD COLUMN consent_at DATETIME;
ALTER TABLE wr_tokens ADD COLUMN consent_version TEXT DEFAULT '';
```

**wr-proxy代码**：

```go
// handleProxy 开头增加同意检查
if token.KnowledgeCaptureEnabled && token.ConsentGiven == 0 {
    writeJSON(w, http.StatusForbidden, map[string]interface{}{
        "error": map[string]interface{}{
            "type":    "consent_required",
            "message": "使用知识捕获功能需先同意隐私协议",
            "consent_url": fmt.Sprintf("/consent?token_id=%d", token.ID),
        },
    })
    return
}
```

### 改造2：员工数据删除入口（P0，1天）

**法律依据**：PIPL第47条（删除权）、第24条（撤回同意权）

**方案**：员工自助+管理员操作双通道

```
Flask后台新增3个API：

1. GET /api/knowledge/my_data?token_id=xxx
   → 查看我的数据：多少条raw、多少条items、多少条memory

2. POST /api/knowledge/delete_my_data
   { "token_id": 5, "scope": "all" }   // all / raw_only / items_only
   → 删除我的所有数据（异步，标记后批量删除）
   → 同时清除相关向量

3. POST /api/knowledge/opt_out
   { "token_id": 5 }
   → 退出知识捕获：设置capture_enabled=0
   → 保留数据但不再捕获新数据
   → 后续可调用delete_my_data彻底清除
```

**wr-proxy配合**：

```go
// handleProxy中检查opt_out状态
if !token.KnowledgeCaptureEnabled || token.OptedOut {
    // 不投递到knowledgeCh
    return
}
```

**删除流程**：

```
员工请求删除
    │
    ▼
标记为"待删除"（status='pending_delete'）
    │
    ▼ 异步批量处理（每小时执行一次）
    │
    ├── wr_knowledge_raw: WHERE token_id=? → DELETE
    ├── wr_knowledge_items: WHERE token_id=? → DELETE
    ├── wr_knowledge_vectors: WHERE item_id IN (已删除items) → DELETE
    ├── wr_agent_memory: WHERE token_id=? → DELETE
    └── wr_knowledge_audit: 记录删除操作
```

### 改造3：raw表数据最小化（P0，0.5天）

**法律依据**：PIPL第5条（不得过度收集个人信息）、第6条（目的限制）

**当前问题**：raw表存储完整prompt和response，即使脱敏后仍可能含隐私信息。

**方案**：捕获时截断 + 提取后清除

```go
// 修改DeliverKnowledge，截断长文本
func DeliverKnowledge(token *Token, reqID, model, endpoint, clientIP string,
    requestBody, responseBody []byte) {
    
    // 截断到最大长度
    maxPromptLen := 500
    maxResponseLen := 1000
    
    promptStr := string(requestBody)
    if len(promptStr) > maxPromptLen {
        promptStr = promptStr[:maxPromptLen] + "...[truncated]"
    }
    
    responseStr := string(responseBody)
    if len(responseStr) > maxResponseLen {
        responseStr = responseStr[:maxResponseLen] + "...[truncated]"
    }
    
    // ... 投递到channel
}
```

**提取后清除**：

```go
// knowledge_extract.go — 提取完成后清除raw表原文字段
func ExtractRawToKnowledge() (processed int, err error) {
    // ... 现有提取逻辑 ...
    
    // 提取成功后，清除raw表中的完整文本
    db.Exec(`UPDATE wr_knowledge_raw SET prompt='', response='', status='done' 
             WHERE id IN (%s)`, processedIDs)
}
```

### 改造4：审计日志（P1，1天）

**法律依据**：数据安全法第27条（数据安全管理）、PIPL第55条（影响评估需记录）

```sql
CREATE TABLE IF NOT EXISTS wr_knowledge_audit (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    action TEXT NOT NULL,           -- capture/extract/access/delete/export/consent/opt_out
    token_id INTEGER,              -- 操作涉及的Token
    item_id INTEGER,               -- 涉及的知识条目ID
    operator_type TEXT DEFAULT 'system',  -- system/admin/user
    operator_id INTEGER,           -- 操作人ID
    detail TEXT,                   -- 操作详情
    ip_address TEXT,               -- 操作来源IP
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_kaudit_action ON wr_knowledge_audit(action);
CREATE INDEX idx_kaudit_token ON wr_knowledge_audit(token_id);
CREATE INDEX idx_kaudit_created ON wr_knowledge_audit(created_at);
```

**记录时机**：

| 事件 | action | 说明 |
|------|--------|------|
| 捕获一条对话 | capture | token_id + 原始请求ID |
| LLM提取知识 | extract | 提取了多少条、什么类型 |
| 有人检索知识 | access | 谁查了什么（MCP/API都记录） |
| 员工请求删除 | delete | 删除范围+执行结果 |
| 知识导出 | export | 导出了什么范围 |
| 员工同意协议 | consent | consent_version |
| 员工退出捕获 | opt_out | 退出时间 |

### 改造5：敏感信息二次检测（P1，2天）

**法律依据**：PIPL第28条（敏感个人信息需单独同意）

现有脱敏规则覆盖6种类型，但不够全面。需要增加：

```
新增检测规则：

1. 人名检测
   规则：常见姓氏(百家姓前100) + 2-3字名字组合
   场景："请帮张伟写一封邮件" → "张伟"可能是人名
   处理：标记为人名，脱敏为[NAME_N]

2. 地址检测
   关键词："住址"/"家庭住址"/"我家在"/"住小区"
   处理：包含地址信息的条目标记sensitivity='restricted'

3. 健康信息检测
   关键词："病历"/"诊断"/"吃药"/"手术"/"住院"/"体检报告"
   处理：包含健康信息的条目标记sensitivity='restricted'，不进入LLM提取

4. 金融账户信息检测
   关键词："银行卡余额"/"贷款金额"/"征信"/"工资条"
   处理：包含金融信息的条目标记sensitivity='restricted'

5. 未成年人信息检测
   关键词："孩子"/"宝宝"/"儿子"/"女儿" + 年龄/学校信息
   处理：PIPL第28条特别保护不满14周岁未成年人信息

6. 私人通信检测
   关键词："帮我写一封给...的信"/"帮我回复...的消息"
   场景：员工用AI工具处理私人通信
   处理：标记为private，不进入知识提取
```

**实现方式**：

```go
// knowledge_filter.go 增加隐私检测
func detectPrivacyRisk(prompt, response string) (riskLevel string, tags []string) {
    tags = make([]string, 0)
    riskLevel = "low"
    
    // 健康信息
    if matchAnyKeyword(prompt+response, healthKeywords) {
        tags = append(tags, "health")
        riskLevel = "restricted"
    }
    
    // 金融信息
    if matchAnyKeyword(prompt+response, financeKeywords) {
        tags = append(tags, "finance_personal")
        riskLevel = "restricted"
    }
    
    // 私人通信
    if matchAnyKeyword(prompt, privateCommKeywords) {
        tags = append(tags, "private_communication")
        if riskLevel != "restricted" {
            riskLevel = "high"
        }
    }
    
    // 未成年人信息
    if matchAnyKeyword(prompt+response, minorKeywords) {
        tags = append(tags, "minor_info")
        riskLevel = "restricted"  // PIPL第28条最严格
    }
    
    return
}
```

**二次检测的处理**：

```
shouldCapture()通过后 → detectPrivacyRisk()
    │
    ├── riskLevel=low → 正常写入raw表
    ├── riskLevel=high → 写入raw表但标记sensitivity='high'
    └── riskLevel=restricted → 不写入raw表，直接丢弃
        （或写入但status='blocked'，30天后自动删除，不进入LLM提取）
```

### 改造6：隐私影响评估报告（P2，运营期）

**法律依据**：PIPL第55条（个人信息保护影响评估）

**评估模板**：

```
一、数据处理概况
  1. 处理目的：从员工与AI助手的对话中提取业务知识
  2. 处理方式：异步捕获→脱敏→规则筛选→LLM提取→存储→检索
  3. 数据类型：对话文本（已脱敏）
  4. 数据规模：预计每日XX条，累计XX条
  5. 存储期限：raw表30天，知识条目按retention_until

二、数据流图
  员工对话 → wr-proxy(脱敏+截断) → raw表 → LLM提取 → items表 → 检索
  所有环节标注安全措施

三、风险识别
  1. 脱敏遗漏风险：6种内置规则+5种新增规则，覆盖率>95%
  2. 存储泄露风险：SQLite文件权限600，仅wr-proxy进程可读
  3. 越权访问风险：部门隔离+敏感度分级+MCP只读
  4. 数据滥用风险：审计日志+操作可追溯

四、保护措施
  1. 技术措施：脱敏→截断→加密存储→权限隔离→审计日志→自动清理
  2. 管理措施：知情同意→删除权→opt_out→影响评估→定期审计
  3. 组织措施：指定个人信息保护负责人→建立投诉渠道

五、评估结论
  □ 风险可控，可继续处理
  □ 需整改后重新评估
  □ 风险过高，停止处理
```

## 8. 两种模式共存

wr-proxy天然支持两种模式共存：

```
模式A：纯路由模式（无合规风险）
  Token.knowledge_capture_enabled = 0
  → wr-proxy只做请求路由+重试+降级
  → 不捕获任何数据
  → 和Hermes/Claude Code直连API效果一样
  → 没有任何法律问题

模式B：知识捕获模式（需要合规）
  Token.knowledge_capture_enabled = 1
  → 需要：consent_given=1 + 同意协议版本匹配
  → 捕获+脱敏+提取+存储
  → 需要满足本文档所有合规要求

默认是模式A，员工主动选择才进入模式B。
这就是 knowledge_capture_enabled 默认=0 的法律意义。
```
