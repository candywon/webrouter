# 企业知识库 — 分期实施计划

> 版本：v1.0 | 日期：2026-05-14

---

## 一期：基础版（1-2周）

### 目标

跑通"捕获 → 存储 → 检索"全流程，验证数据量和噪音比例。

### wr-proxy 侧（Go）

| 任务 | 说明 |
|------|------|
| wr_tokens 表增加字段 | knowledge_capture_enabled + knowledge_department |
| handleProxy 异步投递 | 响应回写后异步投递到 knowledgeCh（带缓冲1024） |
| Knowledge Worker 协程 | 从 channel 消费，信号筛选（规则），写入 raw 表 |
| 信号筛选规则 | 短问答/纯编程/翻译/改写/简单查询 → 丢弃 |
| 定时清理 | raw 表 status=done 超过30天自动删除 |
| MCP端点 | /mcp 路由，initialize + tools/list + tools/call |
| MCP工具(只读) | knowledge_search + knowledge_get_detail + knowledge_list_domains |
| 单域分析引擎 | knowledge_analyze：加载域知识→构建Prompt→转发DeepSeek→返回结果 |
| 模型自动选择 | 按知识量选模型：deepseek-chat(<8万) / gemini-flash(<20万) / gemini-pro(<80万) |
| 分析记录表 | wr_knowledge_analyses，存储任务状态和结果 |

### Flask 侧（Python）

| 任务 | 说明 |
|------|------|
| DB建表 | wr_knowledge_raw + wr_knowledge_items + wr_knowledge_domains + wr_knowledge_analyses |
| 捕获统计API | 今日捕获量/过滤率/待处理数 |
| Token管理扩展 | knowledge_capture_enabled 开关 + knowledge_department |
| 知识列表页面 | 分页 + 按域/部门/类型筛选 |
| 关键词搜索API | LIKE查询，简易版 |
| 分析API | POST /api/knowledge/analyze + GET /api/knowledge/analyze/:task_id |
| 分析结果页面 | 历史分析记录 + 结果查看 |

### 不做

- 不上 LLM 提炼（先用规则筛选观察数据）
- 不上向量/语义检索
- 不上 RAG 注入
- 不上审核队列
- 不上跨域关联分析（一期只做单域）
- 不上分段汇总（一期假设单域<128K）

### 验收标准

1. 指定 Token 开启捕获后，对话能被异步写入 raw 表
2. 信号筛选规则能过滤掉 80%+ 的编程/翻译/短问答
3. 剩余 raw 数据能在 Flask 后台查看
4. 系统对代理性能零影响（P99延迟不变）

## 二期：智能版（2-3周）

### 目标

接入 LLM 提炼，实现数据类知识零幻觉提取 + 语义检索。

### wr-proxy 侧（Go）

| 任务 | 说明 |
|------|------|
| LLM批量评估 | 调用qwen3-coder-flash批量评估知识密度（每批20条） |
| LLM逐条提取 | has_knowledge=true的条目逐条提取factual/analytical/procedural |
| 数据点机器验证 | VerifyDataPoint：value在原文逐字匹配 + 数字一致性检查 |
| 写入 knowledge_items | 通过验证的auto，未通过的pending |

### Flask 侧（Python）

| 任务 | 说明 |
|------|------|
| Embedding向量化 | 调用DashScope text-embedding异步生成向量 |
| 向量存储 | wr_knowledge_vectors 表 |
| 语义搜索API | 余弦相似度 top-k |
| 审核队列页面 | uncertain/pending条目列表 + approve/reject |
| 业务域演化 | pending域自动发现 + 管理员确认/合并 |
| LLM提取管理 | 查看提取结果 + 编辑data_points |

### 提取Prompt调优

需要用真实数据迭代3-5轮：

1. 先跑100条真实raw数据，观察提取质量
2. 统计误提取率、漏提取率、幻觉率
3. 调整Prompt约束 + 规则
4. 再跑500条验证
5. 达标后全量运行

### 验收标准

1. LLM批量评估成本 < 0.02元/100条
2. factual类型数据幻觉率 < 1%（机器验证拦截率>99%）
3. 语义搜索 top-3 准确率 > 80%
4. 审核队列可正常操作

## 三期：增强版（2-3周）

### 目标

RAG自动注入 + 权限隔离 + 企业级管控。

### wr-proxy 侧（Go）

| 任务 | 说明 |
|------|------|
| RAG自动注入 | 用户prompt → embedding → 检索top3 → 注入system prompt |
| 部门权限隔离 | 检索时校验token的department，只返回本部门知识 |
| 敏感度拦截 | sensitivity=restricted的知识不对非管理token返回 |

### Flask 侧（Python）

| 任务 | 说明 |
|------|------|
| 知识统计看板 | 按域/部门/类型的统计图表 + 趋势 |
| 敏感度管控 | 知识条目设置sensitivity + retention_until |
| 保留期限清理 | 过期知识自动归档/删除 |
| 跨部门共享 | 管理员可标记知识为跨部门共享 |
| 导出功能 | 按域/部门导出知识为Excel/JSON |

### 验收标准

1. RAG注入后，相同问题的回答质量提升可感知
2. 部门隔离无越权（A部门token看不到B部门restricted知识）
3. 过期知识自动清理无遗漏
4. 看板数据与实际一致

## 里程碑时间线

```
Week 1-2  一期：基础捕获+规则筛选+SQLite存储+Flask列表/搜索
              ↓ 验证数据量和噪音比例
Week 3-5  二期：LLM提炼+数据零幻觉+Embedding+语义检索+审核队列
              ↓ Prompt调优+幻觉率验证
Week 6-8  三期：RAG注入+权限隔离+敏感度管控+统计看板
              ↓ 企业级完整可用
```

## 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 信号筛选规则不准 | 中 | 噪音多或漏知识 | 一期先观察数据，迭代规则 |
| LLM提取幻觉率超标 | 中 | 数据不可信 | 机器验证兜底，uncertain需人工 |
| Embedding API不可用 | 低 | 语义检索不可用 | 降级到关键词检索 |
| 存储膨胀 | 低 | 磁盘压力 | 定期清理+原文只存锚点 |
| RAG注入降低响应质量 | 中 | 误导回答 | 相似度阈值设高，A/B测试 |
