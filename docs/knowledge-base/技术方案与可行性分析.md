# 企业知识库 — 技术方案与可行性分析

> 版本：v1.0 | 日期：2026-05-14 | 状态：方案设计

---

## 一、需求背景

当企业员工全部通过 wr-proxy 调用 API 处理业务时，API 请求自然将与公司相关的法务合同、财务审计、人力资源招聘、行政考勤、销售售后等业务内容输送给大模型做分析决策，每天产生大量公司知识。

企业知识库的目标：将这些知识分类存放数据库，对外提供检索调用接口。

### 核心定位

```
员工API请求 → wr-proxy(已是流量必经之路) → 捕获+提取 → 分类存储 → 检索调用
```

关键优势：wr-proxy 已经是所有 API 流量的必经之路，天然具备"捕获"能力，不需要额外埋点或改造客户端。

## 二、三个核心难点

### 难点1：知识识别 — 不是简单的查字典

关键词方案的致命问题：

```
"帮我查一下合同到期时间" → 关键词"合同"命中 → 但这只是查询，不是知识
"这份合同的违约金条款和行业惯例有冲突" → 同样命中"合同" → 这才是知识

"Q3毛利率3.2%" → 关键词啥也匹配不到 → 但这是高价值数据知识
"分析一下毛利率" → 关键词匹配到了 → 但这只是个问题，不是知识
```

关键词匹配的是"话题"，不是"知识密度"。真正的分界线是：

**知识 = 包含确定性的信息产出（结论、数据、规则、判断）**

### 难点2：部门归类 — 分类体系必须能随业务演化

传统做法是"先建分类树，再把知识挂上去"。问题：

```
年初：销售部、售后部、财务部
年中：新增跨境电商组（归销售还是独立？）
年末：跨境电商变成事业部，下面又有自己的法务和财务
```

分类体系不能预设，必须从数据中生长出来。

### 难点3：数据类知识零幻觉 — 最关键

LLM 提炼知识的最大风险：

```
原文："Q3毛利率同比收窄3.2个百分点至18.7%"
LLM提取："Q3毛利率下降3.2%"       ← 错误！收窄3.2个百分点≠下降3.2%
LLM提取："Q3毛利率为3.2%"         ← 严重错误！把百分点当成了绝对值

原文："合同违约金为合同总金额的20%"
LLM提取："合同违约金为20%"         ← 丢了关键限定词"合同总金额的"
```

## 三、技术方案

### 3.1 整体架构

```
wr-proxy (Go)
  ├── handleProxy() 现有流程不变
  ├── [新增] 异步捕获 → 写入消息队列(channel/Redis)
  │
  └── Knowledge Worker (Go协程)
        ├── 内容提取（规则 + LLM 双模式）
        ├── 分类打标（部门/业务域/敏感度）
        ├── 向量化（调用 Embedding API）
        └── 写入存储

知识库存储层
  ├── SQLite — 结构化元数据
  ├── 向量索引 — 语义检索
  └── 全文索引 — 关键词检索

Flask管理后台 (Python)
  ├── 知识库管理页面
  ├── 检索API (/api/knowledge/search)
  ├── 分类管理 + 业务域演化
  └── 知识详情 + 审核队列
```

### 3.2 捕获层 — wr-proxy 侧

```go
// 在 handleProxy 的响应回写之后，异步投递
go func() {
    entry := KnowledgeEntry{
        TokenID:     token.ID,
        TokenName:   token.Name,
        Model:       modelName,
        Prompt:      sanitizedPrompt,    // 脱敏后的
        Response:    sanitizedResponse,  // 脱敏后的
        Timestamp:   time.Now(),
        ClientIP:    clientIP,
    }
    knowledgeQueue <- entry  // 带缓冲的channel，不阻塞主流程
}()
```

关键设计：

- **异步**：用带缓冲 channel（如1024），写满则丢弃，绝不影响代理性能
- **脱敏后存储**：复用现有 desensitize 模块，手机号/身份证等已替换为标记
- **可控开关**：Token 级别 `knowledge_capture_enabled` 字段，不是所有人都开

### 3.3 提取层 — 多级漏斗

#### 第1级：信号筛选（零成本，纯规则）

```go
// 基于规则的粗筛
func shouldProcess(entry KnowledgeEntry) bool {
    // 丢弃：单轮短问答（<3轮且响应<200字）
    if len(entry.Conversation) < 3 && len(entry.Response) < 200 {
        return false
    }
    // 丢弃：纯编程类（code block占比>60%）
    if codeBlockRatio(entry.Prompt) > 0.6 {
        return false
    }
    // 丢弃：明确的无知识意图
    if matchNoneKnowledgeIntent(entry.Prompt) {
        return false
    }
    return true
}
```

预计 80%+ 的 API 调用（编程、翻译、改写）在此级被过滤。

#### 第2级：知识密度评估（低成本 LLM，批量处理）

```json
输入：对话全文（每批20条）
输出：{ "has_knowledge": bool, "knowledge_type": "data|conclusion|rule|none", "confidence": 0.85 }

模型：qwen3-coder-flash（最廉价）
成本：约 0.01 元/100条
```

has_knowledge=false 的直接丢弃。

#### 第3级：知识提取（中成本 LLM，逐条处理）

只处理第2级通过的条目。按知识类型分两条路径：

- **factual（事实数据）**：严格锚定原文，不做任何改写
- **analytical/procedural（分析结论/流程规范）**：LLM 可提炼，但必须附原文证据链

### 3.4 数据类知识提取 — 零幻觉方案

#### 核心原则

```
事实/数据类知识 → 只存原文片段 + 位置锚点，LLM 不做任何改写
结论/规则类知识 → LLM 可以提炼，但必须附原文证据链
```

#### 提取 Prompt（严格模式）

```
你是企业数据提取引擎。从以下对话原文中提取事实数据。

【铁律】
1. value字段必须从原文逐字复制，禁止任何改写、换算、推断
2. 如果原文说"收窄3.2个百分点至18.7%"，value只能写"18.7%"，不能写"下降3.2%"
3. 如果数据不确定或有歧义，certainty标为uncertain
4. original_sentence必须包含该数据的完整原句
5. 禁止计算、换算、四舍五入、补零
6. 原文中没有的数据，绝对不能出现在value中

对话原文：
---
{conversation}
---

输出JSON：
{
  "data_points": [
    {
      "metric": "指标名称",
      "value": "从原文逐字复制的值",
      "period": "时间范围（如有）",
      "unit": "单位（如有）",
      "original_sentence": "包含此数据的原文完整句子",
      "certainty": "certain 或 uncertain"
    }
  ],
  "has_no_data": false
}
```

#### 二次机器验证

```go
// 提取完成后，自动验证 value 是否原文中出现
func VerifyDataPoints(dp DataPoint, sourceQuote string) bool {
    // value必须在original_sentence中逐字出现
    if !strings.Contains(dp.OriginalSentence, dp.Value) {
        dp.Certainty = "uncertain"
        dp.Verification = "pending"  // 需人工审核
        return false
    }
    return true
}
```

不管 LLM 怎么提取，只要提取的 value 在原文中找不到精确匹配，就标记为 uncertain，必须人工审核。

### 3.5 分类体系 — 双层归类

#### 第1层：部门归属（由 Token 确定，100%准确）

```
Token "张三-财务部" 产生的知识 → 归属财务部
Token "李四-销售部" 产生的知识 → 归属销售部
不需要LLM猜，天然准确
```

#### 第2层：业务域（LLM 提取 + 管理员确认，可演化）

```
初始预设：legal/finance/hr/admin/sales/service/tech
LLM提取时允许返回 "new_domain" 标记
发现新域 → 入库但标记 pending，需管理员确认
确认后自动加入可选域列表
```

业务域演化机制：

- 当 pending 域积累超过 20 条知识时，管理后台弹出提示
- 管理员可以：确认 / 合并到已有域 / 忽略
- 已确认的域会自动加入后续 LLM 提取的可选列表
- 这样分类体系是从数据中生长出来的，不是人为预设的

### 3.6 存储设计

```sql
-- 原始对话暂存（LLM处理前）
wr_knowledge_raw
├── id INTEGER PRIMARY KEY
├── request_id TEXT          -- 原始请求ID
├── token_id INTEGER
├── token_name TEXT
├── model_name TEXT
├── prompt TEXT              -- 脱敏后的完整prompt
├── response TEXT            -- 脱敏后的完整response
├── turn_count INTEGER       -- 对话轮数
├── status TEXT DEFAULT 'pending'  -- pending/processing/done/skipped
├── created_at DATETIME
└── processed_at DATETIME

-- 知识条目（提取后）
wr_knowledge_items
├── id INTEGER PRIMARY KEY
├── type TEXT                -- factual / analytical / procedural
│   ├── factual: 事实数据（金额、比例、日期、数量）
│   ├── analytical: 分析结论（原因分析、趋势判断）
│   └── procedural: 流程规范（审批流程、操作步骤）
│
├── -- 提取内容 --
├── title TEXT               -- LLM生成的标题（仅用于展示/检索）
├── summary TEXT             -- LLM生成的摘要（仅用于检索，不作为数据源）
├── domain_code TEXT         -- 业务域
├── department TEXT          -- 部门（来自Token）
│
├── -- 原文锚定 --
├── source_request_id TEXT   -- 原始请求ID
├── source_turn_index INTEGER -- 第几轮对话
├── source_quote TEXT        -- 原文片段（精确引用，不改动一字）
├── source_char_start INTEGER -- 原文中的字符起始位置
├── source_char_end INTEGER   -- 原文中的字符结束位置
│
├── -- 数据类专门字段 --
├── data_points TEXT         -- JSON，只用于factual类型
│   [
│     {
│       "metric": "毛利率",
│       "value": "18.7%",
│       "period": "Q3",
│       "unit": "百分比",
│       "original_sentence": "Q3毛利率同比收窄3.2个百分点至18.7%",
│       "certainty": "certain"
│     }
│   ]
│
├── -- 验证 --
├── confidence REAL          -- 0.0-1.0，LLM自评
├── verification TEXT DEFAULT 'auto'  -- auto/pending/verified/rejected
├── verified_by INTEGER      -- 审核人ID
├── verified_at DATETIME
│
├── -- 元数据 --
├── token_id INTEGER
├── token_name TEXT
├── model_name TEXT
├── sensitivity TEXT DEFAULT 'medium'  -- low/medium/high/restricted
├── retention_until DATETIME -- 保留期限
├── created_at DATETIME
└── updated_at DATETIME

-- 业务域管理
wr_knowledge_domains
├── id INTEGER PRIMARY KEY
├── domain_code TEXT UNIQUE
├── domain_name TEXT
├── department TEXT          -- 主要关联部门
├── status TEXT DEFAULT 'pending'  -- active/pending/merged
├── sample_count INTEGER DEFAULT 0 -- 该域下知识条数
├── auto_keywords TEXT       -- JSON，LLM自动提炼的关键词
├── description TEXT
├── merged_into INTEGER      -- 如果被合并到其他域
├── created_at DATETIME
├── confirmed_at DATETIME
└── confirmed_by INTEGER

-- 向量索引（语义检索用）
wr_knowledge_vectors
├── item_id INTEGER PRIMARY KEY  -- 关联wr_knowledge_items.id
├── embedding BLOB               -- 向量，存为JSON float数组
├── model TEXT                    -- 使用的embedding模型
├── created_at DATETIME
```

### 3.8 RAG 自动注入（高级功能）

员工下次问类似问题时，wr-proxy 自动注入相关知识：

```
wr-proxy handleProxy 新增逻辑：
  用户prompt → embedding → 检索知识库 top3
  → 如果相似度 > 阈值，注入system prompt：
  "以下是公司相关知识供参考：\n1. [Q3毛利率分析结论]...\n2. [...]"
  → 再发给上游大模型
```

完全透明，员工不需要手动检索。

## 四、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| **隐私合规** | 员工对话被存储，可能含私人内容 | Token级开关 + 脱敏 + 敏感度分级 + 保留期限自动清理 |
| **性能影响** | 捕获/提取拖慢代理 | 全异步，channel写满则丢，零阻塞 |
| **知识噪音** | 大量无价值对话入库 | 多级漏斗过滤，预计80%+在第1级被丢弃 |
| **存储膨胀** | 向量+原文占用磁盘 | 定期清理过期知识，原文只存hash不存全文（factual存锚点） |
| **数据幻觉** | LLM提炼篡改数据 | 数据类只锚定不提取 + 机器验证（value在原文中逐字匹配） |
| **分类漂移** | 业务域越分越细 | 管理员确认机制 + 合并功能 + 最小样本数阈值 |
| **跨部门越权** | 销售看到财务知识 | 部门级权限隔离，检索API校验token的部门归属 |

## 五、数据流全图

```
API请求/响应
     │
     ▼
wr-proxy handleProxy()
     │
     ├── 正常代理流程（不变）
     │
     └── [异步] 投递到 knowledgeCh (带缓冲1024，满则丢)
              │
              ▼
         Knowledge Worker (Go协程)
              │
              ├── 1. 信号筛选（规则，零成本）
              │     └── 丢弃：短问答 / 纯编程 / 明确无知识意图
              │
              ├── 2. 脱敏处理（复用现有模块）
              │     └── 手机号→[PHONE_1]，身份证→[IDCARD_1]等
              │
              ├── 3. 写入 raw 表（异步，不等待LLM）
              │
              └── 4. 批量提交LLM处理（每5分钟 / 每50条）
                    │
                    ├── 知识密度评估 → has_knowledge=false则丢弃
                    ├── 分类提取 → factual/analytical/procedural
                    ├── 数据点提取 + 原文锚定（factual专有）
                    ├── 二次机器验证（value在原文中逐字匹配）
                    └── 写入 wr_knowledge_items
                         │
                         ├── confidence>=0.8 且 certainty=certain → auto验证
                         ├── certainty=uncertain → pending，等人工审核
                         └── confidence<0.5 → 丢弃

Flask管理后台
     │
     ├── 知识库看板（按部门/域统计）
     ├── 待审核队列（uncertain条目）
     ├── 业务域管理（确认/合并pending域）
     ├── 检索API（关键词 + 语义双模式）
     └── 敏感度管控 + 保留期限设置
```

## 六、可行性评估

| 维度 | 判断 | 依据 |
|------|------|------|
| 捕获 | 完全可行 | wr-proxy 已是流量必经之路，加异步channel即可 |
| 识别 | 可行，需调优 | 多级漏斗 + LLM评估，80%噪音可在第1级滤掉 |
| 归类 | 可行 | 部门由Token确定（零误差），业务域允许演化 |
| 数据准确性 | 可控 | 核心原则：数据只锚定不提取，机器可验证 |
| 性能影响 | 无风险 | 全异步，channel满则丢，不阻塞代理主流程 |
| 存储 | 轻量 | SQLite足够，10万条知识约500MB |
| LLM成本 | 极低 | qwen3-coder-flash批量处理，约0.01元/100条 |
| 部署复杂度 | 低 | 无额外服务依赖，wr-proxy + SQLite即可 |

**唯一需要持续投入的是：提取Prompt的调优 + 机器验证规则的完善。** 这需要跑一段时间真实数据后迭代。

## 七、分期实施计划

### 一期（1-2周）：基础版

- wr-proxy 异步捕获 + 信号筛选 + SQLite存储
- Flask 知识库列表/搜索页面
- 基于关键词的检索API
- 不上 LLM 提炼、不上向量，先跑通流程

### 二期（2-3周）：智能版

- LLM 异步提炼（qwen3-coder-flash 批量处理）
- 数据类知识严格锚定 + 机器验证
- Embedding 向量化 + 语义检索
- 待审核队列 + 业务域演化机制

### 三期（2-3周）：增强版

- RAG 自动注入（wr-proxy 检索 + prompt 增强）
- 部门权限隔离
- 敏感度管控 + 保留期限自动清理
- 知识统计看板 + 趋势分析
