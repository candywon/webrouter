# 企业知识库 — 个人知识库方案

> 版本：v1.0 | 日期：2026-05-19

---

## 1. 一个改动，质的变化

```
原方案：企业知识库（共享）
  员工对话 → wr-proxy捕获 → 提取知识 → 存入共享知识库 → 其他同事可检索
  法律定性：组织处理个人信息 → 受PIPL约束 → 需要知情同意/删除权/审计...

新方案：个人知识库（私有）
  员工对话 → wr-proxy捕获 → 提取知识 → 存入个人知识库 → 仅自己可检索
  法律定性：个人使用工具 → 和Hermes的memory一样 → 基本无合规风险
```

**核心变化**：数据控制者从"企业"变成了"员工自己"。

| 维度 | 企业知识库 | 个人知识库 |
|------|-----------|-----------|
| 谁的数据 | 企业（管理员控制） | 员工个人 |
| 谁能看 | 同部门/跨部门 | 只有自己 |
| 谁决定采集 | 管理员开启 | 员工自己开启 |
| 谁决定删除 | 管理员 | 员工自己 |
| 法律定性 | 组织处理个人信息 | 个人使用工具 |
| 类比 | 公司档案柜 | 个人笔记本 |
| 同类产品 | 无（这是创新的争议点） | Hermes memory / Claude Projects / ChatGPT Memory |

## 2. 法律定性分析

### 2.1 个人使用工具 — 为什么合法

```
PIPL的管辖对象是"个人信息处理者"：
  第4条：个人信息处理者，是指在个人信息处理活动中
        自主决定处理目的、处理方式的组织或个人。

关键判定：
  谁自主决定处理目的和方式？

企业知识库：
  企业管理员决定采集什么、存到哪、谁能看
  → 企业是处理者 → 受PIPL约束

个人知识库：
  员工自己决定是否开启、存储什么、是否删除
  → 员工自己是处理者
  → 员工处理自己的信息（或自己对话中产生的信息）
  → 不构成"处理他人个人信息"
  → PIPL不适用（或适用但自然豁免）
```

### 2.2 与Hermes Memory的对比

| 维度 | Hermes Memory | 个人知识库(wr-proxy) |
|------|--------------|---------------------|
| 谁开启 | 用户自己 | 员工自己(Token开关) |
| 谁存储 | 用户本地 ~/.hermes/ | 服务器DB但按Token隔离 |
| 谁能看 | 只有用户 | 只有该Token持有者 |
| 谁删除 | 用户自己 | 员工自己(删除API) |
| 能否共享 | 不能 | 不能（新方案） |
| 法律定性 | 个人笔记 | 个人笔记 |

**唯一区别**：存储位置不同（本地 vs 服务器）。但存储位置不影响法律定性——云笔记也是个人笔记。

### 2.3 但对话中可能包含他人信息

```
场景：员工A的对话中提到"张三的工资是3万"
  → A的知识库里存了关于张三的信息
  → 这算不算"处理他人个人信息"？

判定：
  1. A在日常对话中得知张三工资 → A已经知道了（不违法）
  2. A把这件事记在个人笔记里 → 个人记录（不违法）
  3. A的个人知识库自动提取了这个信息 → 等同于个人笔记
  4. A没有把这条信息分享给任何人 → 没有传播
  5. 这条信息只存在A的个人空间 → 没有第三方访问

与直接询问"张三工资多少"的区别：
  如果HR系统被A查询 → 组织提供个人信息 → 需要合规
  如果A在日常工作中自然获得并记录 → 个人行为 → 不受PIPL约束

结论：个人知识库存了他人的信息，但只要不共享，
      法律性质等同于"个人记笔记"，风险极低。
```

### 2.4 企业作为"存储服务提供者"的角色

```
问题：个人知识库存在企业的服务器上，企业是什么角色？

类比：
  员工的电子邮件存在企业Exchange服务器 → 企业是"通信服务提供者"
  员工的个人文件存在企业网盘 → 企业是"存储服务提供者"
  员工的个人知识库存在wr-proxy → 企业是"工具+存储服务提供者"

企业义务（与邮箱/网盘相同）：
  1. 不主动查看员工内容（技术保证：Token隔离+加密）
  2. 不将员工内容分享给第三方
  3. 按员工要求删除数据
  4. 保障数据安全（防泄露/防入侵）

企业权利（与邮箱/网盘相同）：
  1. 可以设定存储配额
  2. 可以设定保留期限
  3. 可以在员工离职时清理数据
  4. 可以在法律要求时配合调查

→ 这和提供企业邮箱的法律性质完全一样，已经很成熟了。
```

## 3. 技术方案调整

### 3.1 数据隔离：从部门级到Token级

```
原方案：
  知识条目 → 按部门隔离 → 同部门可见

新方案：
  知识条目 → 按Token隔离 → 仅Token持有者可见

DB变化：
  wr_knowledge_items 增加索引：
  CREATE INDEX idx_kitem_token ON wr_knowledge_items(token_id);
  
  所有查询增加 WHERE token_id = ?
  MCP/RAG/检索API全部按token_id过滤
```

### 3.2 检索隔离

```go
// 原方案：部门级过滤
func searchKnowledge(query string, department string) []KnowledgeItem {
    // WHERE department = ? OR sensitivity = 'low'
}

// 新方案：Token级过滤
func searchKnowledge(query string, tokenID int) []KnowledgeItem {
    // WHERE token_id = ?
    // 绝对隔离，不存在跨Token访问
}
```

### 3.3 RAG注入隔离

```go
// 原方案：注入同部门知识
ragContext := retrieveRAGContext(prompt, token.Department, ...)

// 新方案：只注入自己的知识
ragContext := retrieveRAGContext(prompt, token.ID, ...)
```

### 3.4 MCP工具隔离

```
MCP工具调用时：
  Authorization: Bearer sk-wr-xxxx
  → 解析Token → token_id=5
  → knowledge_search 只返回 token_id=5 的知识
  → knowledge_analyze 只能分析自己的知识
  → memory_recall 只能看自己的记忆
```

### 3.5 管理员角色变化

```
原方案：管理员可以查看所有知识、审核条目、合并域
新方案：管理员只能看统计信息（总量/存储占用），不能看具体内容

管理员可见：
  ✓ 全局统计：总条目数、存储占用、捕获率
  ✓ 资源管理：Token配额、存储配额
  ✓ 系统健康：捕获延迟、提取队列

管理员不可见：
  ✗ 具体知识条目内容
  ✗ 员工的原始对话
  ✗ 员工的记忆内容
  ✗ 员工的检索历史

这和企业邮箱管理员一样：
  管理员能看"邮箱占用2GB"，不能看"邮件内容"。
```

## 4. 保留"可选共享"机制

完全不共享虽然最安全，但会丧失一些价值。可以设计**可选的、员工主动发起的**共享机制：

```
模式1：纯个人（默认）
  所有知识仅自己可见
  与Hermes memory完全一致

模式2：主动共享（可选）
  员工手动选择某条知识 → 点击"共享到部门"
  → 该条知识复制到部门共享区
  → 其他部门成员可检索
  → 这时候该条知识变成"组织知识"，需要PIPL合规

模式3：团队空间（可选）
  员工创建团队 → 邀请同事加入
  → 团队内共享知识
  → 每个成员主动加入 → 构成"同意"
```

### 4.1 主动共享的合规逻辑

```
员工A把一条知识共享到部门：
  1. A点击"共享" → 弹出确认框：
     "共享后，该知识将对[财务部]的5名同事可见。
      请确认该知识不包含敏感个人信息。
      ☐ 我确认可以共享"
  2. A确认 → 该条知识复制到部门共享区
  3. 共享行为记录审计日志

法律逻辑：
  A是这条知识的"数据主体"（对话是A产生的）
  A主动选择共享 → 构成"同意"
  A共享的是业务知识，不是"张三工资3万"这类他人信息
  → 合规

边界情况：
  如果A共享了"张三工资3万" → A的行为，不是企业的行为
  但系统应该检测并提示"该条目可能包含他人个人信息，确认共享？"
```

### 4.2 技术实现

```sql
-- wr_knowledge_items 增加可见性字段
ALTER TABLE wr_knowledge_items ADD COLUMN visibility TEXT DEFAULT 'private';
-- private: 仅Token持有者可见（默认）
-- department: 本部门可见（需主动共享）
-- team: 指定团队可见
-- public: 全公司可见（需管理员审核）

-- 共享记录
CREATE TABLE IF NOT EXISTS wr_knowledge_shares (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    item_id INTEGER NOT NULL,
    shared_by_token_id INTEGER NOT NULL,  -- 谁共享的
    shared_to_type TEXT NOT NULL,          -- department/team/public
    shared_to_id TEXT DEFAULT '',          -- 部门名/团队ID
    consent_text TEXT DEFAULT '',          -- 确认文本
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (item_id) REFERENCES wr_knowledge_items(id)
);
```

## 5. 法律风险对比

| 场景 | 企业知识库(共享) | 个人知识库(私有) | 个人库+可选共享 |
|------|-----------------|-----------------|----------------|
| 默认采集 | 需同意(PIPL) | 自己开=同意 | 自己开=同意 |
| 员工不知情 | 违法 | 不可能(自己开) | 不可能 |
| 跨人访问 | 默认存在 | 不存在 | 员工主动触发 |
| 删除权 | 需提供(PIPL) | 自己删 | 自己删 |
| 敏感信息 | 企业担责 | 个人行为 | 共享时提示 |
| 管理员偷看 | 可能 | 技术隔离 | 技术隔离 |
| 开发方风险 | 中高 | 极低 | 低 |
| 与同类产品比 | 无先例 | 和Hermes一样 | 和GitHub私有仓库类似 |

## 6. 前端调整

### 6.1 知识库页面变化

```
原方案（企业知识库）：
  [捕获统计] [原始对话] [知识条目] [审核队列] [业务域] [单域分析] [分析记录] [知识提取] [搜索]

新方案（个人知识库）：
  [我的知识] [我的记忆] [知识搜索] [全域分析] [共享管理]

变化：
  - "审核队列"去掉（个人知识无需审核）
  - "业务域"弱化（个人视角，不是管理员视角）
  - "共享管理"新增（管理自己共享出去的知识）
  - 整体从"管理后台"风格改为"个人知识助手"风格
```

### 6.2 新增共享操作

```
每条知识条目旁边：
  [共享到部门] [共享到团队] [设为公开]

点击后弹出确认：
  ┌─────────────────────────────────┐
  │  共享知识                        │
  │                                 │
  │  标题：Q3毛利率数据              │
  │                                 │
  │  共享范围：                      │
  │  ○ 本部门（财务部，5人）         │
  │  ○ 指定团队                     │
  │  ○ 全公司                       │
  │                                 │
  │  ⚠️ 共享后其他同事将能看到此知识  │
  │  ⚠️ 请确认不包含他人敏感信息      │
  │                                 │
  │  ☐ 我确认此知识可以安全共享      │
  │                                 │
  │      [取消]    [确认共享]        │
  └─────────────────────────────────┘
```

## 7. 合规改造简化

原方案需要6项合规改造，新方案大幅简化：

| 改造项 | 企业知识库 | 个人知识库 | 说明 |
|--------|-----------|-----------|------|
| 知情同意机制 | P0 必须 | **不需要** | 自己开启=同意 |
| 员工数据删除入口 | P0 必须 | **需要但极简** | 一键清空个人数据 |
| raw表数据最小化 | P0 必须 | **建议但非必须** | 个人数据个人负责 |
| 审计日志 | P1 必须 | **简化** | 只记录共享操作 |
| 敏感信息二次检测 | P1 必须 | **共享时才需要** | 私有不检测，共享时提示 |
| 隐私影响评估 | P2 必须 | **不需要** | 个人使用工具无需评估 |

**P0从3项降到0项**，合规工作量从5.5天降到1天。

## 8. 开源开发方风险再评估

```
个人知识库方案下，开发方风险进一步降低：

1. 帮助侵权 → 极低
   工具提供的是"个人笔记"功能，不是"组织监控"功能
   和提供笔记本应用一样，没人因为Evernote被用于记隐私信息而起诉Evernote

2. 产品责任 → 极低
   脱敏bug仍存在，但影响范围仅限个人
   个人对自己的数据负责

3. 刑法 → 无
   开发方不处理任何个人信息

4. License合规 → 不变

5. 跨境传输 → 无
```

**开发方风险从"中高"降到"极低"，和Hermes/Notion/Obsidian等个人工具同等水平。**

## 9. 方案对比总结

| 维度 | 企业知识库 | 个人知识库 | 个人库+可选共享 |
|------|-----------|-----------|----------------|
| 合规风险 | 高 | 极低 | 低 |
| 合规工作量 | 5.5天 | 1天 | 2天 |
| 开发方风险 | 中高 | 极低 | 低 |
| 知识价值 | 高（跨人共享） | 中（仅个人） | 高（可选共享） |
| 用户体验 | 被动采集 | 主动掌控 | 主动掌控+可选分享 |
| 同类产品 | 无先例 | Hermes memory | GitHub仓库(private+可公开) |
| 法律定性 | 组织处理个人信息 | 个人使用工具 | 个人使用+自愿分享 |
| 管理员可见 | 全部 | 仅统计 | 仅统计+共享审核 |

**推荐方案：个人知识库 + 可选共享**

理由：
1. 默认个人 → 法律风险极低，和Hermes/Claude Memory同等
2. 可选共享 → 保留跨人协作价值
3. 员工主动掌控 → 用户体验好，不产生被监控感
4. 开发方风险极低 → 开源无忧
5. 和GitHub私有仓库模式一致（默认私有，主动公开）——用户已熟悉此模式
