# 企业知识库 — 超长上下文分析设计

> 版本：v1.0 | 日期：2026-05-19

---

## 1. 需求本质

现有两种模式对比：

| 模式 | 检索（已有） | 全域分析（新增） |
|------|-------------|-----------------|
| 类比 | 图书馆查几本书 | 把整个书架搬来通读 |
| 输入 | query关键词 | 整个业务域的全部知识 |
| 输出 | 几条相关知识 | 分析报告/趋势/洞察/交叉验证 |
| 上下文 | <10K tokens | 100K~1M tokens |
| 典型问题 | "Q3毛利率是多少？" | "财务域所有数据有什么趋势和矛盾？" |

核心场景：

```
管理者："帮我把财务域所有知识过一遍，看看数据之间有没有矛盾"
       → 把finance域全部423条知识灌给DeepSeek（1M上下文）
       → 大模型做交叉验证、趋势分析、异常发现

员工："总结一下我们法务域的合同风险点"
       → 把legal域201条知识灌给大模型
       → 大模型做全局归纳、风险提炼

管理员："对比一下销售域和售后域的知识，客户反馈和销售话术有没有脱节？"
       → 把sales+service两个域的知识一起灌入
       → 大模型做跨域关联分析
```

## 2. 可用大模型上下文窗口

| 模型 | 上下文 | 价格(输入/输出 per M tokens) | 适合场景 |
|------|--------|------------------------------|----------|
| DeepSeek-V3/R1 | 128K | ¥0.009/0.027 | 中等域分析，性价比最高 |
| DeepSeek-Chat(long) | 128K | 同上 | 同上 |
| Gemini 2.5 Pro | 1M | ¥0.16/0.48 | 超大域全量分析 |
| Gemini 2.0 Flash | 1M | ¥0.005/0.015 | 超大域低成本分析 |
| Qwen-Max | 1M | 需确认 | 阿里云生态 |

## 3. 知识量→Token估算

单条知识平均token消耗：

```
标题:    ~20 tokens
摘要:    ~80 tokens
数据点:  ~50 tokens × 平均2个 = ~100 tokens
原文锚点: ~200 tokens
元数据:  ~30 tokens
─────────────────────────
单条合计: ~430 tokens（约400-500）
```

| 域 | 预估条数 | Token消耗 | 需要的模型 |
|----|---------|-----------|-----------|
| 单个小域(service) | ~100条 | ~43K | DeepSeek 128K 绰绰有余 |
| 单个中域(legal) | ~200条 | ~86K | DeepSeek 128K 够用 |
| 单个大域(finance) | ~400条 | ~172K | 需Gemini 1M 或分段分析 |
| 两个域合并 | ~600条 | ~258K | 需Gemini 1M |
| 全域(8个域) | ~1500条 | ~645K | 需Gemini 1M 或 Qwen-Long |
| 极端全量 | ~3000条 | ~1.3M | 需Qwen-Long 10M |

## 4. 架构设计

```
智能体/用户发起全域分析请求
         │
         ▼
   MCP工具: knowledge_analyze
   或 REST API: POST /api/knowledge/analyze
         │
         ▼
   wr-proxy 知识分析引擎
         │
         ├── 1. 确定范围（域/部门/时间/类型）
         ├── 2. 从DB加载知识条目
         ├── 3. Token预算评估
         │     ├── 总token < 目标模型窗口70% → 单次提交
         │     └── 总token > 目标模型窗口70% → 分段提交+汇总
         ├── 4. 构建分析Prompt
         ├── 5. 通过wr-proxy自身转发到大模型
         │     （复用现有路由+重试+降级逻辑）
         ├── 6. 返回分析结果
         └── 7. 分析记录入wr_knowledge_analyses表
```

关键设计：**wr-proxy自己就是API网关，分析请求可以复用自己的路由能力**。不需要直连某个Provider，而是走wr-proxy的标准代理流程，享受智能路由+重试+降级。

## 5. 分析Prompt模板

### 5.1 单域分析

```
你是企业知识分析引擎。以下是从公司{domain_name}知识库中提取的全部{count}条知识。

请完成以下分析任务：
1. 【数据交叉验证】检查数据点之间是否存在矛盾或不一致
2. 【趋势发现】识别数据中的趋势、规律和异常
3. 【风险预警】标注潜在的风险点或需要关注的问题
4. 【知识缺口】指出该域中明显缺失的知识领域
5. 【摘要总结】用5-8句话概括该域的核心状况

【关键约束】
- 分析结论必须基于提供的知识内容，不得臆造
- 引用数据时必须标注来源条目ID
- 如果知识之间存在矛盾，明确指出并分析可能原因
- 不确定的分析结论标注[待验证]

知识内容：
---
{knowledge_items_formatted}
---

输出JSON：
{
  "cross_validation": [
    { "finding": "发现", "item_ids": [42, 67], "type": "conflict|consistency|anomaly" }
  ],
  "trends": [
    { "trend": "趋势描述", "evidence_ids": [12, 34, 56], "confidence": 0.8 }
  ],
  "risks": [
    { "risk": "风险描述", "severity": "high|medium|low", "evidence_ids": [78] }
  ],
  "gaps": [
    { "area": "缺失领域", "importance": "high|medium|low" }
  ],
  "summary": "5-8句话的核心概括"
}
```

### 5.2 跨域关联分析

```
你是企业跨域知识分析引擎。以下包含来自{domain_count}个业务域的知识。

请完成以下跨域分析：
1. 【跨域关联】不同域的知识之间是否存在因果/影响关系
2. 【数据矛盾】跨域数据是否有不一致（如销售说增长，售后说投诉增多）
3. 【协同机会】不同域之间可能的协作改进点
4. 【全局风险】需要多个域协同应对的风险

知识内容：
---
{knowledge_items_formatted}
---
```

### 5.3 分段分析+汇总（大域超窗口时）

```
第一阶段（分段）：
  对每个分段独立分析，输出分段结论

第二阶段（汇总）：
  你是企业知识汇总引擎。以下是对{domain_name}域{count}条知识的{segment_count}个分段分析结果。
  请综合所有分段结论，输出统一的全域分析报告。
  
  分段结论：
  ---
  {segment_summaries}
  ---
```

```

Content-Type: application/json
Authorization: Bearer sk-wr-xxxx
```

```json
{
  "domains": ["finance"],
  "departments": [],
  "types": [],
  "date_from": "2026-01-01",
  "date_to": "2026-05-19",
  "analysis_type": "domain_overview",
  "model": "deepseek-chat",

  "options": {
    "include_cross_validation": true,
    "include_trends": true,
    "include_risks": true,
    "include_gaps": true

}
```

| 字段 | 类型 | 必填 | 说明 |

| analysis_type | string | 否 | domain_overview/cross_domain/custom |

| max_budget_tokens | int | 否 | 最大消耗token预算，默认128000 |

## 7. 模型自动选择策略

```go
func selectAnalysisModel(totalTokens int, budgetTokens int) string {
    // 根据知识总量自动选模型
    switch {
    case totalTokens < 80000:   // <8万token
        return "deepseek-chat"          // 128K窗口，最便宜
    case totalTokens < 200000:  // 8-20万token
        return "gemini-2.0-flash"       // 1M窗口，低成本
    case totalTokens < 800000:  // 20-80万token
        return "gemini-2.5-pro"         // 1M窗口，高质量
    default:                    // >80万token
        return "qwen-long"              // 10M窗口，极端场景
    }
}
```

选择逻辑：

| 知识量 | Token估算 | 自动选模型 | 原因 |
|--------|-----------|-----------|------|
| <200条 | <86K | deepseek-chat | 128K窗口够用，最便宜 |
| 200-500条 | 86K-215K | gemini-2.0-flash | 1M窗口，成本极低(0.005/M) |
| 500-2000条 | 215K-860K | gemini-2.5-pro | 1M窗口，高质量分析 |
| >2000条 | >860K | qwen-long | 10M窗口，分段汇总 |

## 8. 分段策略（超大域）

当知识总量超过目标模型窗口70%时，自动分段：

```
域: finance, 423条, 估算182K tokens
目标模型: deepseek-chat (128K窗口, 有效90K)
→ 超出！自动分段

分段1: 条目1-210, ~90K tokens → 独立分析 → 分段结论1
分段2: 条目211-423, ~92K tokens → 独立分析 → 分段结论2

汇总: 分段结论1+2, ~5K tokens → deepseek-chat汇总 → 最终报告
```

分段原则：
1. 按时间排序分段（每个分段是连续时间段的知识）
2. 每段控制在模型窗口的60-70%
3. 汇总阶段用同一模型，输入量极小

## 10. 分析记录存储

```sql
CREATE TABLE IF NOT EXISTS wr_knowledge_analyses (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id TEXT UNIQUE NOT NULL,
    token_id INTEGER NOT NULL,
    token_name TEXT DEFAULT '',
    
    -- 分析范围
    domains TEXT NOT NULL,              -- JSON数组 ["finance"]
    departments TEXT DEFAULT '[]',      -- JSON数组
    types TEXT DEFAULT '[]',            -- JSON数组
    date_from TEXT DEFAULT '',
    date_to TEXT DEFAULT '',
    item_count INTEGER DEFAULT 0,       -- 分析的知识条数
    analysis_type TEXT DEFAULT 'domain_overview',
    
    -- 执行信息
    model_used TEXT DEFAULT '',
    strategy TEXT DEFAULT '',           -- single_pass / segmented / segmented_summary
    segment_count INTEGER DEFAULT 1,
    tokens_consumed INTEGER DEFAULT 0,
    cost REAL DEFAULT 0.0,
    duration_ms INTEGER DEFAULT 0,
    
    -- 结果
    status TEXT DEFAULT 'processing',   -- processing/completed/failed
    result TEXT DEFAULT '',             -- JSON，分析结果
    error_message TEXT DEFAULT '',
    
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    completed_at DATETIME
);

CREATE INDEX idx_kanalyses_token ON wr_knowledge_analyses(token_id);
CREATE INDEX idx_kanalyses_status ON wr_knowledge_analyses(status);
CREATE INDEX idx_kanalyses_created ON wr_knowledge_analyses(created_at);
```

## 11. 安全与权限

| 风险 | 对策 |
|------|------|
| 跨部门分析泄露知识 | 分析范围受Token部门权限限制，只能分析本部门可见域 |
| 一次性灌入过多token导致费用失控 | max_budget_tokens预算上限 + 分析前预估确认 |
| 大模型对超长上下文"遗忘"中间内容 | 分段策略，每段独立分析后汇总 |
| 分析结论被当成确定性知识 | 分析结果不写入知识库，仅作为参考报告存储 |

**分析结果不入知识库** — 分析是大模型的推理输出，不是锚定原文的事实数据。如果分析发现了有价值的新洞察，需要走正常的知识提取流程（LLM评估→验证→入库），不能直接入库。
