# 企业知识库 — 智能体增强方案全景

> 版本：v1.0 | 日期：2026-05-19

---

## 0. 问题本质

让AI智能体"更懂公司"，本质是三个不同层面的问题：

```
层面1：知道什么 — 事实知识（Q3毛利率18.7%、合同审批流程5步）
层面2：记住什么 — 历史记忆（上次讨论的方案、用户偏好、项目上下文）
层面3：能做什么 — 行动能力（查知识库、发邮件、改代码）
```

不同层面需要不同的增强方式：

| 层面 | 需求 | MCP能解决吗 | 更好的方式 |
|------|------|------------|-----------|
| 知道什么 | 专业知识注入 | 部分能（检索时才有） | System Prompt预注入 / RAG自动注入 |
| 记住什么 | 跨会话持久记忆 | 不能 | 持久记忆层 / 对话摘要压缩 |
| 能做什么 | 主动获取知识 | 能 | MCP + Web搜索 + API直调 |

---

## 1. 六种增强方式对比

| 方式 | 原理 | 触发时机 | 知识时效性 | 实现复杂度 | 适用场景 |
|------|------|---------|-----------|-----------|---------|
| **① System Prompt预注入** | 在system prompt中直接写死知识 | 会话开始前 | 静态，更新需改配置 | 极低 | 公司简介、固定流程、角色设定 |
| **② RAG自动注入** | 检测用户问题，自动检索并注入相关知识 | 每次请求时 | 实时 | 中 | 专业术语解释、制度查询、数据引用 |
| **③ MCP工具调用** | AI自主决定何时调用知识库工具 | AI判断需要时 | 实时 | 中 | 主动深度查询、分析、对比 |
| **④ 持久记忆层** | 跨会话存储关键信息，下次自动加载 | 每次会话开始时 | 持久累积 | 高 | 用户偏好、项目上下文、长期跟踪 |
| **⑤ 对话摘要压缩** | 长对话压缩为摘要，保留关键信息 | 对话超出窗口时 | 会话内 | 中 | 长项目讨论、多轮分析 |
| **⑥ 知识微调** | 用公司数据微调模型 | 训练时 | 训练截止 | 极高 | 行业术语、公司特有概念 |

## 2. 逐方案详解

### ① System Prompt 预注入

**原理**：在发给大模型的 messages 最前面加一条 system role 消息，包含公司知识。

```
wr-proxy handleProxy 改造：
  用户请求 → 注入system prompt → 转发给上游大模型
```

**注入模板**：

```
你是{company_name}的AI助手。以下是公司核心知识供你参考：

## 公司基本信息
{company_profile}

## 业务域概览
{domain_summaries}

## 常用流程
{key_procedures}

## 注意事项
- 引用数据时必须标注来源
- 不确定的信息要明确说明
- 涉及金额/比例等精确数据时，优先从知识库检索验证
```

**实现方式**（wr-proxy侧）：

```go
func injectSystemPrompt(messages []Message, token *WRToken) []Message {
    // 1. 从缓存获取该部门的知识摘要（每5分钟刷新一次）
    summary := getDepartmentKnowledgeSummary(token.KnowledgeDepartment)
    
    // 2. 构建system prompt
    systemContent := fmt.Sprintf(knowledgePromptTemplate, 
        companyName, summary.Profile, summary.Domains, summary.Procedures)
    
    // 3. 插入到messages最前面
    systemMsg := Message{Role: "system", Content: systemContent}
    return append([]Message{systemMsg}, messages...)
}
```

**优势**：
- 零额外API调用，不增加延迟
- 每次对话都能获取到基础知识
- 实现极简，wr-proxy改几行代码

**局限**：
- 占用context窗口（通常2-5K tokens）
- 信息量有限，不可能把全部知识写进去
- 更新不实时（依赖缓存刷新周期）

**适合注入的内容**：
- 公司简介（50-100字）
- 部门职能概述（每部门1-2句）
- 关键流程的标题和入口提示（不要写完整流程，提示AI可以调用工具查）
- 常见术语表（10-20个）

**不适合注入的内容**：
- 具体数据（会过时）
- 完整流程（太长）
- 敏感信息（每次请求都发）

### ② RAG 自动注入

**原理**：wr-proxy 在转发用户请求前，先对prompt做语义检索，把高相关度的知识片段自动注入。

```
用户prompt → embedding → 检索知识库top3 → 注入system prompt → 转发
```

**与MCP的核心区别**：

| | RAG自动注入 | MCP工具调用 |
|--|-----------|-----------|
| 触发者 | wr-proxy自动 | AI主动决定 |
| 用户感知 | 完全透明 | 可能感知AI在调工具 |
| 延迟 | +检索时间（50-200ms） | +检索+AI决策时间 |
| 精度 | 可能注入无关内容 | AI按需精准查询 |
| 适用 | 常见问题自动增强 | 深度查询和分析 |

**实现方式**（wr-proxy侧）：

```go
func handleProxyWithRAG(w http.ResponseWriter, r *http.Request) {
    // ... 现有认证、脱敏等流程 ...
    
    // RAG自动注入
    if token.KnowledgeCaptureEnabled && token.RAGEnabled {
        ragContext := retrieveRAGContext(
            userPrompt,           // 用户当前prompt
            token.KnowledgeDepartment,
            3,                    // top-k
            0.7,                  // 最低相关度阈值
        )
        
        if ragContext != "" {
            // 注入到system prompt
            ragSystem := fmt.Sprintf(
                "以下是从公司知识库自动检索到的相关信息，供你参考：\n%s\n请在回答时参考以上信息，但不要直接复制，用你自己的理解重新组织。如果以上信息与问题不相关，忽略即可。",
                ragContext,
            )
            messages = injectRAGSystemPrompt(messages, ragSystem)
        }
    }
    
    // ... 继续正常代理流程 ...
}
```

**关键设计**：
1. **相关度阈值**：只有相似度>0.7才注入，避免噪音
2. **精简格式**：每条知识只注入title+summary+关键data_point，不注入全文
3. **不透明注入**：用户不知道有RAG，体验上感觉AI"就是懂"
4. **部门权限**：只检索本部门可见知识

**RAG注入示例**：

```
用户问："Q3的销售情况怎么样？"

自动检索到3条相关知识：
  1. Q3销售数据 | 营收4.28亿元，同比增长15.3% | 相似度0.89
  2. Q3客户增长 | 新增客户127家 | 相似度0.72
  3. Q3退货率 | 退货率3.2%，环比下降0.5% | 相似度0.71

注入后AI回答（不需要额外调用工具）：
"根据公司数据，Q3销售情况良好：营收4.28亿元，同比增长15.3%，
 新增客户127家，同时退货率3.2%环比下降了0.5个百分点。"
```

**RAG vs MCP如何选择**：
- 常见问题（80%）→ RAG自动注入，零延迟感知
- 复杂查询（20%）→ MCP工具，AI深度检索
- 两者可以共存：RAG兜底，MCP按需

### ③ MCP 工具调用（已设计）

详见[检索API与MCP集成设计](检索API设计.md)，这里不再重复。

补充一个关键点：**MCP工具可以串联**。

```
用户："帮我查一下合同编号CT-2024-089的违约金条款，然后看看和行业惯例有没有冲突"

AI执行流程：
  1. knowledge_search({ query: "CT-2024-089 违约金", domain: "legal" })
     → 找到合同知识条目，违约金为合同总金额的20%
  
  2. knowledge_search({ query: "违约金 行业惯例 比例", domain: "legal" })
     → 找到行业惯例知识，一般为10-15%
  
  3. AI综合分析：该合同违约金20%高于行业惯例10-15%，建议法务审核
```

这种多步推理是RAG做不到的，只有MCP能让AI自主决策查询策略。

### ④ 持久记忆层

**原理**：把跨会话的关键信息持久化存储，每次新会话开始时自动加载。

```
会话1: 用户说"我在负责跨境电商项目，团队5人"
  → 提取记忆: {项目: 跨境电商, 角色: 负责人, 团队规模: 5}

会话2: 用户问"帮我写个项目周报"
  → 自动加载记忆 → AI知道是跨境电商项目的周报
```

**存储设计**：

```sql
CREATE TABLE IF NOT EXISTS wr_agent_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    token_id INTEGER NOT NULL,
    memory_type TEXT NOT NULL,        -- fact/preference/context/project
    key TEXT NOT NULL,                -- 如 "current_project"
    value TEXT NOT NULL,              -- 如 "跨境电商"
    source TEXT DEFAULT '',           -- 来源：对话摘要 / 用户明确告知 / 系统推断
    confidence REAL DEFAULT 1.0,      -- 0-1
    access_count INTEGER DEFAULT 0,   -- 被引用次数
    last_accessed_at DATETIME,
    expires_at DATETIME,              -- 过期时间（项目记忆可能在项目结束后失效）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(token_id, memory_type, key)
);
```

**记忆类型**：

| 类型 | 示例 | 生命周期 |
|------|------|---------|
| fact | 公司名、部门名、产品线 | 永久 |
| preference | "回复用中文"、"不要用代码解释" | 永久 |
| context | "我在负责XX项目"、"本季度重点是降本" | 项目/季度结束 |
| project | "项目用的是Vue3+FastAPI" | 项目结束 |

**提取方式**：

```go
// 在handleProxy响应回写后，异步提取记忆
func extractMemory(prompt string, response string, tokenID int) {
    // 用廉价LLM提取关键信息
    // Prompt: "从以下对话中提取需要长期记住的关键事实、偏好、上下文信息"
    // 模型: qwen3-coder-flash
    
    memories := llmExtractMemories(prompt, response)
    
    for _, m := range memories {
        // upsert: 存在则更新，不存在则插入
        upsertMemory(tokenID, m.Type, m.Key, m.Value, m.Confidence)
    }
}
```

**加载方式**：

```go
// 在注入system prompt时，加载该用户的记忆
func loadMemoryContext(tokenID int) string {
    memories := queryActiveMemories(tokenID)
    
    var sb strings.Builder
    sb.WriteString("以下是关于当前用户的已知信息：\n")
    for _, m := range memories {
        sb.WriteString(fmt.Sprintf("- %s: %s\n", m.Key, m.Value))
    }
    return sb.String()
}
```

**与知识库的区别**：

| | 知识库 | 持久记忆 |
|--|-------|---------|
| 存储 | 公司级共享知识 | 用户/Token级私有信息 |
| 来源 | 对话内容提取+验证 | 对话中的偏好/上下文 |
| 验证 | 严格锚定+机器验证 | 宽松，允许推断 |
| 权限 | 部门级隔离 | Token级隔离 |
| 目的 | 事实准确性 | 个性化/连续性 |

### ⑤ 对话摘要压缩

**原理**：长对话超出context窗口时，把早期对话压缩为摘要，保留关键信息。

```
原始对话（20轮，50K tokens）
  │
  ▼ 前15轮压缩为摘要（2K tokens）
  │
  ▼ 摘要 + 最近5轮完整对话 → 继续对话
```

**实现方式**（wr-proxy侧）：

```go
func compressConversation(messages []Message, maxTokens int) []Message {
    totalTokens := estimateTokens(messages)
    
    if totalTokens <= maxTokens {
        return messages  // 不需要压缩
    }
    
    // 1. 保留最近N轮完整对话
    recentCutoff := len(messages) - 6  // 保留最近3轮(user+assistant)
    recentMessages := messages[recentCutoff:]
    olderMessages := messages[:recentCutoff]
    
    // 2. 压缩早期对话为摘要
    summary := summarizeConversation(olderMessages)
    
    // 3. 构建压缩后的对话
    summaryMsg := Message{
        Role: "system",
        Content: fmt.Sprintf("以下是之前对话的摘要：\n%s", summary),
    }
    
    return append([]Message{summaryMsg}, recentMessages...)
}
```

**与知识库的协同**：

对话压缩时，可以同时把有价值的内容提取到知识库：

```
长对话压缩
  │
  ├── 摘要 → 作为上下文继续对话
  └── 有价值的片段 → 投递到知识库捕获channel
```

### ⑥ 知识微调（长期方向）

**原理**：用公司数据微调开源模型，让模型"内化"公司知识。

**与RAG/MCP的区别**：

| | RAG/MCP | 知识微调 |
|--|---------|---------|
| 知识在哪 | 外部数据库，运行时注入 | 模型权重内部，训练时嵌入 |
| 更新方式 | 实时 | 重新训练 |
| 精确性 | 高（原文锚定） | 低（可能产生幻觉） |
| 成本 | 检索成本 | 训练成本（GPU+数据标注） |
| 适用 | 可变知识 | 稳定的行业术语/概念体系 |

**当前不建议投入**，原因：
1. webrouter定位是API网关，不是模型训练平台
2. 微调需要大量标注数据，当前知识库还没建设
3. RAG+MCP已经能解决90%的问题

**未来可能的方向**：
- 用知识库数据做LoRA微调，让模型更好地理解公司术语
- 微调后的模型部署为webrouter的专属channel

## 3. 组合方案

单一方式都有局限，实际效果最好的是组合使用：

### 方案A：基础增强（一期推荐）

```
System Prompt预注入（静态知识）
    + RAG自动注入（常见问题自动增强）
```

- 实现成本低
- 用户零感知
- 覆盖80%的常见问题

### 方案B：智能增强（二期推荐）

```
方案A
    + MCP工具调用（深度查询/分析）
    + 持久记忆层（个性化/连续性）
```

- 覆盖95%场景
- AI有了"查"和"记"的能力
- 支持复杂多步推理

### 方案C：完整增强（三期推荐）

```
方案B
    + 对话摘要压缩（长对话不丢失）
    + 超长上下文全域分析（深度洞察）
```

- 全面覆盖
- 长短期记忆兼备
- 支持全域级别分析

1. 在wr_tokens增加 `system_prompt_knowledge TEXT` 字段
2. handleProxy在转发前插入system message
3. Flask后台增加配置入口

## 8. 安全红线

| 规则 | 说明 |
|------|------|
| System Prompt不含敏感数据 | 只注入公开信息，敏感数据走MCP按需查询 |
| RAG注入受部门权限限制 | 自动检索只返回本部门可见知识 |
| 记忆层Token隔离 | A用户的记忆B用户绝对看不到 |
| MCP调用有频率限制 | 防止AI过度调用消耗资源 |
| 对话摘要不含原文 | 摘要是LLM改写的，不作为数据源 |
