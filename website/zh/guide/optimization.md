# 优化特性

## 概述

wr-proxy 内置请求优化功能，在请求到达上游 Provider 前自动转换——降低 Token 成本、提升缓存命中率、优雅处理长对话。

所有功能通过系统设置控制，可独立开关，无需重启 wr-proxy。

## 功能开关

| 功能 | 设置键 | 默认值 | 说明 |
|------|--------|--------|------|
| 动态内容重排 | `feature_dynamic_content_last` | 关闭 | 将动态内容移至上下文末尾 |
| Token 压缩 | `feature_token_compression` | 关闭 | 压缩超长 system prompt |
| 会话压缩 | `feature_session_compression` | 关闭 | 摘要长对话历史 |

在管理面板 **系统设置** 或 `PUT /api/settings` 中切换。

## 1. 动态内容重排

### 问题

当动态内容（URL、时间戳、哈希值）出现在对话前部时，会阻止上游 prompt 缓存生效——即使核心指令未变，每次请求看起来都"不同"。

### 解决方案

wr-proxy 检测包含动态模式的消息，并在其 role 组内重排——将动态消息推到末尾：

| 模式 | 示例 |
|------|------|
| URL | `https://api.example.com/v1/...` |
| 日期 | `2026-06-07` |
| 时间 | `14:30:00` |
| 哈希/UUID | `a1b2c3d4e5f6...` |
| 长数字 | `1234567890123` |

```
重排前：                          重排后：
[system: "你是..."]               [system: "你是..."]
[user: "检查 https://..."]        [user: "状态怎么样？"]
[user: "状态怎么样？"]              [user: "检查 https://..."]  ← 移到末尾
```

效果：静态前缀在上游保持可缓存。

## 2. Token 压缩（RTK）

### 问题

超长 system prompt（2000+ 字符）每次请求都消耗 Token，即使指令几乎不变。

### 解决方案

启用后，超过阈值的 system prompt 被原地压缩：

```
原始（3000 字符）：
"你是一位专业代码审查员。遵循以下准则：
 1. 检查安全漏洞...
 2. 验证错误处理...
 ... (还有 2000 字符) ...
 99. 确保命名一致..."

压缩后：
"你是一位专业代码审查员。遵循以下准则：
 1. 检查安全漏洞...
 2. 验证错误处理...

[... 2300 字符省略 ...]

 98. 使用类型提示。
 99. 确保命名一致。"
```

模型收到紧凑表示，同时保留关键的首尾指令。

## 3. 会话压缩

### 问题

长对话（10+ 轮）累积显著 Token 成本。早期消息可能与当前话题无关。

### 解决方案

当对话超过压缩阈值时，wr-proxy：

1. 完整保留最近 5 轮
2. 将早期轮次摘要为简洁的系统消息
3. 在最近消息前注入摘要

```
压缩前（15 轮）：                    压缩后：
[system: "你是..."]                 [system: "你是..."]
[第1轮: user/assistant]             [system: "[第1-10轮摘要]
[第2轮: user/assistant]              共 10 轮对话，约 5000 字符。
 ...                                 主要话题：部署流程、数据库配置..."]
[第10轮: user/assistant]            [第11轮: user/assistant]  ← 保留
[第11轮: user/assistant]            [第12轮: user/assistant]
 ...                                ...
[第15轮: user/assistant]            [第15轮: user/assistant]
```

## 会话上下文感知

除了三个可开关的功能外，wr-proxy 的智能模型选择也会考虑会话上下文：

- **大上下文**（累积 >16K tokens）：复杂度评分 +0.15，可能触发 premium 模型选择
- **中等上下文**（>8K tokens）：复杂度评分 +0.08

确保对话越深入，模型选择越适配。

## API 兼容性

三项优化功能同样支持 Anthropic（`/v1/messages`）和 Cohere（`/v1/chat`）API 格式，在优化前先进行格式转换。

## 管理端点

| 端点 | 说明 |
|------|------|
| `GET /admin/features` | 查看当前功能开关状态 |
| `POST /admin/features` | 从数据库重新加载功能开关 |
| `POST /admin/conversation_compress` | 手动触发会话压缩 |
